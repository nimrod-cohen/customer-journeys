## What is being built

A **serverless, multi-tenant marketing CDP** on AWS: companies get isolated workspaces, ingest behavioral events over HTTP, maintain unified customer profiles with rolling aggregates, define segments, and reach people via broadcasts (one-off blasts) and campaigns (multi-step journeys) through Amazon SES. An admin SPA provides segment/campaign builders, a WYSIWYG email editor, dashboards, and per-workspace cost views.

## Tech stack (all decisions are locked — see §0)

- **Backend:** TypeScript on Node.js 20+, AWS Lambda (real Node runtime, full npm incl. `sharp`).
- **HTTP entry:** API Gateway **REST API** (not HTTP API — chosen for request validation, per-workspace API keys + usage plans, native WAF).
- **Database:** Supabase (managed PostgreSQL), pooled connection from Lambda. System of record.
- **Queue:** SQS **FIFO** (+ DLQ), `MessageGroupId = profile_id`.
- **Email:** Amazon SES; each workspace sends from its own verified domain.
- **Auth:** Supabase Auth; validated at the gateway by a **Lambda authorizer** (REST API has no native JWT authorizer).
- **Email editor:** a CUSTOM in-house designer (`web/src/email-designer/`, ported from the owner's nomentor builder; Preact + @preact/signals) — this deliberately OVERRIDES the spec's GrapesJS choice. The surviving invariant is unchanged: the editor **emits MJML, never hand-rolled HTML** (design JSON → `mjml-serializer.ts` → server `compileMjml`, strict). Templates are a LIBRARY; attaching one to a broadcast/campaign CLONES it (`kind='copy'`, `source_template_id`) into an independently editable working copy.
- **IaC:** AWS CDK (TypeScript).
- **Frontend:** Preact/React SPA, Vite, static on S3 + CloudFront. **Server-calling buttons auto-lock:** the shared kit `Button` (`web/src/ui/kit.tsx`) shows a spinner + disables itself whenever its `onClick` returns a Promise — so wire every server-calling button so its handler RETURNS the promise (`onClick={save}` / `onClick={() => save(id)}`, never `() => void save()`), and pass `loading` explicitly for `type="submit"` buttons. This is a standing rule (no double-submits, clear in-flight state).
- **Testing:** Vitest or Jest; Supabase CLI / Testcontainers for ephemeral Postgres; `aws-sdk-client-mock` for AWS SDK; LocalStack for the thin E2E tier.

## Non-negotiable invariants

These are the properties the whole system exists to guarantee. Every change must preserve them; they are pass/fail acceptance gates (§18), not aspirations.

1. **Tenant isolation.** Every tenant-scoped row carries `workspace_id NOT NULL`. Isolation is defense-in-depth: (a) app code filters by `workspace_id` in *every* query, (b) Postgres RLS for user-context (admin app) connections. **Critical caveat (§3):** backend processing Lambdas connect with the Supabase **service role, which bypasses RLS** — they MUST scope by `workspace_id` in code; RLS is not their safety net.

2. **Workspace is never client-supplied.** Derive it from the API key at ingest (§7) and from the authorizer-injected claim on the admin API (§12). Never trust a `workspace_id` in a request payload.

3. **Per-profile event ordering.** `MessageGroupId = profile_id` gives per-profile FIFO. Code must be **idempotent and order-convergent** so a `progress` arriving before `profile_created` still resolves correctly (upsert a stub). Don't unit-test "the queue delivered in order" — that's SQS's job; test your convergence.

4. **No lost events.** `200` is returned only after SQS accepts (durable boundary). Forced failures retry, then go to DLQ — never vanish.

5. **Idempotency.** `events.event_id` is producer-supplied and the dedupe key (`INSERT ... ON CONFLICT DO NOTHING`); `outbox.dedupe_key` and broadcast `(broadcast_id, profile_id)` prevent double-sends.

6. **Segment SQL is compiled, never interpolated.** The rule-AST → SQL compiler (§8) whitelists fields/operators, emits parameterized SQL only, and **always** prepends `workspace_id = $ws`. This is security-critical and the single highest-value unit-test target.

7. **Sending is gated on verification.** The Dispatcher refuses to send for any workspace not `active`/verified (§10). Every send passes suppression → frequency-cap → quiet-hours checks. **Verification source (overrides §10A):** the gate (`canSend`) is fed from a VERIFIED row in `sending_domains` — the per-domain model's source of truth — derived in `services/dispatcher/src/dispatch.ts` (a verified domain also supplies the `no-reply@<domain>` From fallback). The legacy `workspaces.sending_identity` jsonb is only a fallback for old data; the per-domain onboarding flow never populates it, so gating on it alone silently refused fully-verified workspaces (regression test: `services/dispatcher/test/verified-domain-gate.integration.test.ts`).

## Roles (§3A)

Four roles. `system-admin` (in `platform_admins`) is the **only cross-tenant role** — the authorizer injects an `is_platform_admin` claim, RLS has a narrow exception for it, and every cross-tenant access is written to `admin_audit_log`. The other three (`owner` / `marketer` / `accounting`) are workspace-scoped via `workspace_users.role`; a user may hold different roles in different workspaces.

## Architecture flow

```
Producer --API key--> API GW REST (validate, usage plan, WAF)
  --> Ingest Lambda (resolve workspace from key, upsert profile) --> SQS FIFO (group=profile_id)
  --> Processor (idempotent, workspace-scoped): events -> profile_features -> re-eval segments -> membership diff + change_log
        -> segment entry drives campaign enrollment (§9B) and is available to broadcasts (§9A)
  --> outbox row -> 2nd SQS queue --> Dispatcher: suppression -> freq-cap -> quiet-hours -> SES (workspace config set) -> messages_log
SES events -> SNS -> Feedback Lambda -> per-workspace suppression + reputation policing (auto-suspend offender)
```

Broadcasts (§9A) and campaigns (§9B) both emit sends through the **same** outbox → Dispatcher → SES pipeline. Campaigns are a **table-driven state machine** over `campaign_enrollments`, swept by a scheduled Campaign-runner Lambda (chosen over Step Functions: cheaper for long multi-day waits at this scale).

**Campaign trigger kinds + enrollment (§9B, phase 3):** a campaign's `definition.startNode` is its single **trigger node** — one of THREE `kind`s (`services/campaign-runner/src/dsl.ts`):
- **`segment_entry`** — the trigger segment lives on the **campaign ROW** (`campaigns.trigger_segment_id` + `trigger_on` enter|exit), NOT on the node. Entering/leaving that segment drives `enrollFromSegmentChange`.
- **`event`** — the node carries `eventType` (REQUIRED) + an optional **payload filter** (`filter`, a `payload.*`-namespace `AstNode`); BOTH live in the **definition JSON** (no migration). An ingested event of that type drives `enrollFromEvent`. The filter is a **pure, closed-grammar, in-memory** eval over the event payload (`evaluateEventPayloadFilter` in `core.ts` — reuses the `@cdp/segments` operator whitelist via `resolveOperator`; a non-whitelisted operator or a non-`payload.*` field THROWS — never interpolated).
- **`manual`** — no auto-source; enrolled by the API (`POST /campaigns/:id/enroll`).

**Where enrollment fires (same workspace-scoped tx as segment re-eval, no nested BEGIN/COMMIT):** right after the event row + feature recompute + realtime segment re-eval, on the SAME tx client. **PROD:** `services/processor` `runPlanInWorkspaceTx → runSegmentReevalInTx` (the `SegmentReeval` marker now also carries `eventType`/`eventPayload`/`eventId`) invokes `enrollFromSegmentChange` per entered/exited delta AND `enrollFromEvent`. **DEV mirror:** `services/local-api` `recomputeFeaturesAndSegments` (called by `sendProfileEvent` + the merge path) fires the same two; `sendProfileEvent` also calls `enrollFromEvent` directly with the just-recorded event. (Before phase 3, `enrollFromSegmentChange` was exported but NOT invoked by any live path — phase 3 wired it into the live flow.) **Manual/API:** `POST /campaigns/:id/enroll` (`enrollIntoCampaign`, capability `manage_content`) takes **exactly one** of `profile_id` | `segment_id`, scopes the campaign/profile/segment to `ctx.workspaceId` (a cross-workspace id → 404, inv.2), and enrolls the profile or a **point-in-time SEGMENT SNAPSHOT** (current members via `buildResolveAudience` — both dynamic + manual sources; later segment changes do NOT retroactively enroll). `workspace_id` is from the authorizer ctx ONLY — a body `workspace_id` is ignored.

**Re-enrollment policy = `'once'` for ALL THREE kinds (the documented default).** Enforced structurally by `UNIQUE(campaign_id, profile_id)` + `INSERT … ON CONFLICT DO NOTHING` (`buildEnrollmentInsert`) and decided in code by `decideReenrollment(hasExisting, 'once')` (`DEFAULT_REENROLLMENT_POLICY` in `core.ts`). All three kinds funnel through `buildEnrollmentInsert`, so a replayed event / repeated segment entry / re-run manual enroll inserts at most one enrollment. `'always'` (re-entry) stays a forward-compat enum value but is NOT enabled (no migration; the policy is a code default — if ever persisted it would be a single `campaigns.reenrollment_policy` column). No sends are emitted by enrollment (enrollment writes `campaign_enrollments` only; the runner sweep later drives sends through the Dispatcher).

## How to build (until tooling exists, this is the contract)

- **Build tenancy first** (workspaces, `workspace_id` everywhere, RLS, authorizer) before any feature, so nothing is retrofitted.
- **Work phase by phase (§17)**; a phase is done only when its §18 acceptance criteria pass as tests in CI.
- **Test-first (§16A):** write the phase's §18 criteria as failing tests, then implement to green.
  - **Don't mock Postgres** in the integration tier — ordering, idempotency, and isolation bugs live in the SQL and RLS. Use local Supabase (Supabase CLI) or Testcontainers.
  - **Do mock SES** — assert `SendEmail` is called with the right Configuration Set, only after suppression/cap checks; never send real mail.
- Local dev: SAM local / serverless-offline + LocalStack (SQS/S3/SNS/API GW) + Supabase CLI. Seed multiple workspaces with overlapping `external_id`s/emails to test isolation.

### Status of scaffolding

**Company → Workspace hierarchy (extends spec §6):** a `companies` table groups workspaces — every `workspace` has `company_id NOT NULL` (migration `0012`). Tenant **isolation is unchanged**: it stays at the workspace level (every tenant row carries `workspace_id`; RLS still keys on `workspace_id`). A company is purely the organizational parent so a platform admin can pick a company → a workspace within it (sidebar `CompanyWorkspacePicker`, fed by `GET /admin/companies`). Workspaces created without a `company_id` (chiefly in integration tests) are auto-assigned to a shared `Unassigned` company by a BEFORE-INSERT trigger (migration `0013`); the product's own creation paths (registration, `POST /workspaces`) always supply a real `company_id`, so `Unassigned` never appears from normal use. **A user belongs to ONE company:** access is per-workspace (`workspace_users`), but `addMember` rejects adding a user to a workspace whose company differs from one they're already in (409) — so a user's memberships always stay within a single company (which may own several workspaces).

**Registration creates a COMPANY only — never a workspace (the owner creates the first one manually).** `POST /auth/register` (`registerOwner`, local-api `session.ts`) inserts the owner `users` row + a `companies` row tagged with `owner_user_id` (migration `0032`) — NO workspace, NO `workspace_users`. The minted token is logged-in but workspace-less, so the response carries `needs_workspace: true` + the `company`. Because the production `authorize()` core hard-denies a non-platform-admin token with no active workspace, the workspace-less owner can't reach any capability-gated route (not even `/me`); so creating the first workspace is a **session route** `POST /workspace/bootstrap` (`createFirstWorkspace`) — like `/workspace/switch`, it authenticates the token directly, resolves the company server-side from `owner_user_id = <token sub>` (never client-supplied, inv.2), inserts the workspace + owner `workspace_users`, and **re-mints the token** with the new active workspace. `devLogin` mirrors this: a workspace-less owner who owns a company logs in 200 with `needs_workspace` (instead of the old 403); anyone else with no membership is still 403. SPA: `App.tsx` routes a logged-in non-admin with `!workspaceId` to the `CreateFirstWorkspace` screen (session `needsWorkspace`); `restoreSession` skips `/me` in that state; `bootstrapWorkspace()` enters the shell. Additional workspaces (for an owner already in one) are still created via the capability-gated `POST /workspaces` in Company settings.

**Identity key (overrides spec §6/§7):** a profile's per-workspace identity/merge key is **`email`**, NOT `external_id`. Events arrive from many source systems; email is the only identifier that stitches a person's events together, so ingestion requires `email`, profiles are `UNIQUE(workspace_id, email)` (migration `0010`), and the ingest/processor upserts + the manual `POST /profiles` all merge on `(workspace_id, email)`. `external_id` is optional metadata (the spec's original dedupe-by-external_id design is superseded).

**Sending domains are a LIST with a per-domain setup screen (extends §10):** a workspace can have several sending domains — `sending_domains` table (migration `0021`, workspace-scoped + RLS), each with a `verified` flag. UI is `/onboarding` (the **list only** — `SendingDomainsList`, nav label "Sending domains") → click a row or "Add domain" opens the **per-domain setup screen** `/onboarding/:id` (or `/onboarding/new`) — `SendingDomainDetail`. There you save the domain (pending), see its **Amazon SES Easy-DKIM CNAME records** to publish, and **verify via SES** (`POST /sending-domains/:id/check`). Verification uses **real SES**: on first open each domain is provisioned as an SES email identity (`SesEmailClient.createDomainIdentity` → 3 DKIM tokens, stored in `sending_domains.ses_identity` + `dkim_tokens`, migration `0023`); the check calls `getIdentityVerificationAttributes` and verifies ONLY when `DkimStatus === 'SUCCESS'` — no manual flip. **The DKIM CNAME target host is read FROM SES, never constructed** (multi-region: each company picks its own SES region): SES returns `DkimAttributes.SigningHostedZone` (region-specific, e.g. `dkim.il-central-1.amazonses.com`), persisted in `sending_domains.signing_hosted_zone` (migration `0029`) and used verbatim as `<token>.<signing_hosted_zone>` (`dkimCnameHost` in `handlers.ts`; pre-`0029` rows backfill on next load). Fallbacks only if SES omits it / local mock: `dkim.<region>.amazonses.com`, then `dkim.amazonses.com`. **SES is PER-COMPANY:** each company stores its own AWS SES credentials (`company_ses_config` table, migration `0024`: region + access_key_id + secret_access_key; secret is write-only over the API, never returned; company-scoped RLS like `companies`). Managed in **Company settings** (`CompanySesConfig`, API `GET/PUT/DELETE /company/ses-config`, capability `manage_sending_domain`). The sending-domain handlers build the SES client from the active workspace's company config (`sesForWorkspace` → `createSesClient` from `@cdp/email`); **with no company config they fall back to the local MOCK** SES, so dev/tests verify deterministically and the domain setup screen shows a "verification is simulated" note (`sesConfigured:false`). (The earlier `USE_REAL_SES` env switch and the `0022` TXT-token approach were superseded; the `verify_token` column was dropped in migration `0030`.) A `domain_senders` row (named "From" identity, migration `0020`) may only be created for a **verified** domain (enforced in `createDomainSender`) and is managed **inside that domain's setup screen** (local-part input fixed to `@<that-domain>`), never in a global list. API (all `manage_sending_domain`): `GET/POST /sending-domains`, `GET /sending-domains/:id` (domain + DNS records), `POST /sending-domains/:id/check`, `DELETE /sending-domains/:id` (blocked while it has senders); `GET/POST /domain-senders`, `DELETE /domain-senders/:id`. (The legacy workspace-level wizard routes `/sending-domain/{start,check,activate}` were removed — the per-domain screen above replaces them. The onboarding cores `startDomain`/`checkDomain`/`activate` in `@cdp/service-onboarding` remain, exercised by the acceptance/integration tiers.) **`GET /domain-senders` is `manage_content`** (read needed by the broadcast/campaign From dropdown); creating/deleting a sender stays `manage_sending_domain`.

**From / To / Subject live on the EMAIL INSTANCE (a broadcast/campaign's working copy), NOT on a library template (migration `0028`):** `email_templates` gained `subject text`, `sender_id uuid → domain_senders(id) ON DELETE SET NULL`, and `to_address text NOT NULL DEFAULT '{{customer.email}}'`. A **library template is just a reusable DESIGN and shows NO envelope** in the editor — the From/To/Subject Card (`email-sender` / `email-to` / `email-subject`) renders ONLY when the editor is editing an email *instance* (`TemplateEditor`'s `instance` = opened via a broadcast/campaign return flow OR `kind='copy'`). Attaching a template to a broadcast/campaign **clones** it into a working copy (`kind='copy'`), and that copy's envelope is edited in the editor (autosaved with the design); `cloneTemplate` copies these columns. **Editor return context** (`store/editorReturn.ts`) is persisted in `sessionStorage` so a page refresh inside the editor keeps the contextual "← Back to broadcast/campaign" target (it used to fall back to the template library); standalone editor opens (TemplatesList "New template"/Edit, the "Design email" header shortcuts) call `clearEditorReturn()` so a stale return can't mislabel Back. The **To** is a recipient token (default `{{customer.email}}`) rendered per recipient at send (falls back to `profile.email`; suppression/unsubscribe still key on `profile.email`). **From / To / Subject are ALL MANDATORY to send a broadcast — there is NO no-reply fallback (overrides the earlier `from_selected` design).** The **From** must be a real named sender (a verified-domain `domain_senders` row, `email_templates.sender_id`); the editor's From dropdown has only a disabled "Choose a sender…" placeholder + the named senders (NO "No-reply" option), and shows a hint to add one under Sending domains when the workspace has none. `sendBroadcast` joins the template and 409s in order: "Choose who the email is from…" when `sender_id` is null, "Set the To field…" when `to_address` is blank, "Add a subject line…" when `subject` is blank — *then* the verified-domain gate. The **Dispatcher** reads `compiled_html, subject, sender_id, to_address` from the template, resolves `sender_id` → `"Name" <email>` (`fromAddress` in `core.ts`, which still has a `no-reply@<domain>` last-resort for a null `sender_id` — broadcasts can't reach it), and renders `to_address`. The broadcast/campaign **outbox payload no longer carries** subject/sender — only `broadcast_id`. `cloneTemplate` copies `subject`/`sender_id`/`to_address`/`from_selected` onto the working copy (so attaching a configured library template yields a sendable copy); the e2e seed creates a `domain_senders` row and points the seeded template's `sender_id` at it so seeded-template clones stay sendable. `validateSenderId` (local-api, reused by `createTemplate`/`updateTemplate`) rejects a cross-workspace `sender_id` (inv.2); `GET /domain-senders` is `manage_content` (the editor's From dropdown). The broadcast wizard / campaign builder no longer have envelope fields — they point you to "Design email". **The wizard also gates its Content step on the envelope:** it fetches the attached copy's `{subject, sender_id, to_address}`, computes `missingEnvelope` (From/To/Subject), and `step1Valid` stays false until all three are set — so `wizard-next` and the later breadcrumbs (`canReach`) are disabled and an `email-incomplete` hint lists what's missing (`email-complete` once done). The user can't reach Schedule/Send without a deliberately-chosen From. (`from_selected` migration `0034` remains in the schema but is no longer the gate — `sender_id` presence is.) **Broadcast Content step (instance model):** until an email exists the step offers "Start from a template" (a picker that CLONES the chosen library template into the broadcast's own copy on select) OR "Start from a blank design" (`design-email` → editor, `createAs:'copy'`). Once a copy is attached the picker is GONE — the step shows the instance (`email-instance`) with **Edit email** (`design-email`) and **Start over** (`email-replace`, which just unreferences the copy). The template is only a starting point; you cannot swap the underlying template inline. Returning from the editor lands on the **Content** step (the editor records `markReturnedTo`; the wizard restores the step even when no copy was saved). The **Schedule** step offers **Send now** (the default — the finish button SENDS immediately via `POST /broadcasts/:id/send`, surfacing a send failure rather than leaving a silent draft) or **Schedule for a date & time** (saves a `scheduled` broadcast), plus a separate **Save as draft** action (`wizard-save-draft` → persists with no send time, status `draft` — "created but not sent or scheduled"). Scheduling also offers a **timezone** (`schedule-tz`, default the browser zone, full IANA list via `Intl.supportedValuesOf`): the wall-clock time is interpreted in that zone → UTC (`zonedInputToUtcIso`, DST-correct) and the chosen zone is persisted in `broadcasts.scheduled_tz` (migration `0033`) so edit round-trips it and the list shows the send time in that zone (`whenLabel(ts, tz)`). **A scheduled send must be ≥ 5 minutes from now** (`MIN_SCHEDULE_LEAD_MS`, enforced in BOTH places): the wizard sets the picker's `min`, disables `wizard-save`, and shows a `schedule-too-early` hint when the chosen instant is sooner; the server's `createBroadcast`/`updateBroadcast` re-check via `scheduleLeadError` and 400 ("A broadcast must be scheduled at least 5 minutes from now."). The step **breadcrumbs are clickable** — `canReach(target)` lets you jump to any step whose prerequisites are met instead of clicking Next. **Row actions on the broadcasts list are consolidated into one kebab (⋮) `ActionMenu`** (`broadcast-actions` trigger; reusable component in `web/src/ui/kit.tsx`): Edit/Send/Delete only for unsent (draft/scheduled), Duplicate always; an async item keeps the menu open + spins itself + locks the others until it settles (the standing server-button rule). Each item keeps its original `data-testid` (`broadcast-edit`/`broadcast-duplicate`/`send-broadcast`/`broadcast-delete`) — e2e tests just open the menu first. **Branches CONVERGE into a single trunk — a rejoin diamond (phase 6b, v0.35.0, extends §9B):** inserting an If/branch no longer mints a second exit — its arms **OPEN and REJOIN** a single continuing trunk. `insertOnEdge`'s condition branch on A→B now sets **BOTH `onTrue` AND `onFalse` to `B`** (the continuation), so B becomes the **JOIN** purely STRUCTURALLY — *the node with 2+ incoming edges* — there is **NO stored `join` flag** (honors the single-graph, no-extra-model invariant). The trunk continues below the join. **Each arm has its OWN `+`** (CampaignCanvas already renders one `campaign-edge-insert` per outgoing edge): clicking an arm's `+` inserts a node BETWEEN the If and the join via the existing single-out path (`If.onTrue→New→join`); an **EMPTY arm** is simply the If slot still pointing straight at the join (it passes through). To **terminate one arm**, insert an `exit` on that arm edge — `insertOnEdge('exit')` (and the convenience `insertExitOnArm`) now allows it whenever the arm is CONVERGING (`reachableWithoutEdge` — the join stays reachable via the other arm) and still REFUSES an exit insert that would orphan B on a NON-converging single-out edge. `deleteNode` on a condition collapses to the shared continuation (the join survives because it's reachable via the onTrue survivor). **No migration** (`definition` is jsonb) and the **runner is UNCHANGED** (it just follows `current_node` through the join). Layout/connectors were already diamond-aware (longest-path depth, `recenterJoins`, `subtreeWidth` counts a shared join once, down-only `computeEdges`, axis-aligned `orthogonalPath`); the one addition: when two edges from the SAME source converge on the SAME target (an empty-arm diamond), `computeEdges` **lane-offsets only their SOURCE x** (`ARM_LANE`) so the two arm connectors + their `+` controls don't stack — they still END at the identical join top-center, so the arms visibly CONVERGE on one node. The e2e tokenizes connector path endpoints and asserts ≥2 paths land on the same `(x,y)`. The seeded "Welcome journey" is now a converging diamond (`cond.onTrue→sendY→join`, `cond.onFalse→join`, `join→exit`). (b) a **`webhook`** action kind on the action node (`url` http(s) ONLY; `method` ∈ GET/POST/PUT/PATCH/DELETE; optional `headers` object, `bodyTemplate` string, positive `timeoutMs`, non-negative integer `maxRetries`; `next` edge). **Phase 2 (v0.30.0) makes the runner EXECUTE both** (see the runner-execution bullet below) — they are no longer no-ops.
- **Runner executes `hour_of_day_window` + `webhook` (tz-aware, §9B, phase 2):**
  - **Workspace tz governs all time math.** The tick reads `workspaces.settings->>'timezone'` (default `UTC`, validated via `isValidTimeZone`) FROM the enrollment's `workspace_id` (never client/per-broadcast) and threads it into `processNode`. **`hour_of_day_window`**: inside the allowed window/day (ws tz) → **advance immediately**; outside → **PARK** with `next_run_at = nextWindowOpening(now, win, tz)` (the next opening converted ws-tz wall-clock → UTC, DST-correct via the existing two-pass `zonedInputToUtcIso`; overnight wrap, inclusive-through-:59 end, and `daysOfWeek` keyed on the window-OPENING day all handled). The `next_run_at` gate is honored by the REAL sweep SQL (no app timer). `wait`/`wait-until` is also tz-aware now: a **bare wall-clock** `until` (no Z/offset) is interpreted in the ws tz; an explicit-offset/Date `until` is honored verbatim. New shared helpers `zonedComponents` / `isWindowOpen` / `nextWindowOpening` live in `packages/shared/src/timezone.ts`.
  - **Webhook fires POST-COMMIT, never inside the FOR UPDATE tx** (mirrors `enqueueSends`): chainTick collects a `{kind:'webhook'}` side effect (intent), the tx commits the guarded advance, THEN `runWebhooks` calls the **injected `WebhookHttpClient`** (`@cdp/runner-webhook`; prod = a fetch-based client with an AbortController timeout, tests inject a fake — NEVER a real host). An external call must not hold the row lock. Single-winner FOR UPDATE ⇒ the webhook is **AT-MOST-ONCE**; a per-`(campaign,profile,node)` `activity_log` marker (`dedupe_key`, partial unique index, migration `0036`, `ON CONFLICT DO NOTHING`) prevents a crash-recovery re-fire.
  - **`@cdp/runner-webhook`** (new package): `assertWebhookTargetAllowed` (the SSRF/allowlist guard) + `executeWebhook` (pure, over the injected client) + `fetchWebhookClient`. **Safety model:** the guard runs BEFORE any client call — **deny-by-default per-workspace host allowlist** (`workspaces.settings->'webhook_allowlist'`, an array of hosts; empty/missing ⇒ every host refused; NEVER client-trusted, inv.2) PLUS a literal-IP/host refusal (loopback/`localhost`, `169.254/16` metadata, RFC1918, IPv6 ULA/loopback) and **http(s)-only**. A blocked target → `{ok:false,error:'blocked',attempts:0}` with ZERO client calls, never throws. **Retry policy:** bounded by `1+maxRetries`, retry only on ≥500 / network-timeout, NEVER on 4xx. The body is merge-rendered (`{{customer.*}}`, email-parity) and an encrypted auth-header secret is decrypted at call time only (`@cdp/db` secret-crypto) — the wire carries plaintext, the stored definition + the recorded `activity_log` detail never do. **Failure isolation = continue-on-failure:** a failed/blocked webhook is a notification side effect, not a gate — it records an `activity_log` row (`source='webhook'`, outcome `success`/`failed`/`blocked`) and the enrollment ADVANCES to `node.next` (never stranded on a flaky host); the outcome is visible in Activity.
- **`validateCampaignDefinition` is tightened to the builder invariants** (the DSL stays the single graph model): in addition to the existing exactly-ONE-trigger / resolvable-edges / reachable-exit / per-type-field checks, it now **rejects CYCLES** (`detectCycle` — a DFS grey-stack; a self-loop or a back-edge to a node on the current recursion stack throws, but a diamond / re-convergence is ALLOWED) and **rejects ORPHANS** (`findOrphans` — every defined node must be reachable from `startNode`). Cycle detection runs BEFORE the reachable-exit check so a looping graph is diagnosed as a cycle rather than a generic "no exit".
