## What is being built

A **serverless, multi-tenant marketing CDP** on AWS: companies get isolated workspaces, ingest behavioral events over HTTP, maintain unified customer profiles with rolling aggregates, define segments, and reach people via broadcasts (one-off blasts) and automations (multi-step journeys) through Amazon SES. An admin SPA provides segment/automation builders, a WYSIWYG email editor, dashboards, and per-workspace cost views.

## Tech stack (all decisions are locked ŌĆö see ┬¦0)

- **Backend:** TypeScript on Node.js 20+, AWS Lambda (real Node runtime, full npm incl. `sharp`).
- **HTTP entry:** API Gateway **REST API** (not HTTP API ŌĆö chosen for request validation, per-workspace API keys + usage plans, native WAF).
- **Database:** Supabase (managed PostgreSQL), pooled connection from Lambda. System of record.
- **Queue:** SQS **FIFO** (+ DLQ), `MessageGroupId = profile_id`.
- **Email:** Amazon SES; each workspace sends from its own verified domain.
- **Auth:** Supabase Auth; validated at the gateway by a **Lambda authorizer** (REST API has no native JWT authorizer).
- **Email editor:** a CUSTOM in-house designer (`web/src/email-designer/`, ported from the owner's nomentor builder; Preact + @preact/signals) ŌĆö this deliberately OVERRIDES the spec's GrapesJS choice. The surviving invariant is unchanged: the editor **emits MJML, never hand-rolled HTML** (design JSON ŌåÆ `mjml-serializer.ts` ŌåÆ server `compileMjml`, strict). Templates are a LIBRARY; attaching one to a broadcast/automation CLONES it (`kind='copy'`, `source_template_id`) into an independently editable working copy.
- **IaC:** AWS CDK (TypeScript).
- **Frontend:** Preact/React SPA, Vite, static on S3 + CloudFront. **Server-calling buttons auto-lock:** the shared kit `Button` (`web/src/ui/kit.tsx`) shows a spinner + disables itself whenever its `onClick` returns a Promise ŌĆö so wire every server-calling button so its handler RETURNS the promise (`onClick={save}` / `onClick={() => save(id)}`, never `() => void save()`), and pass `loading` explicitly for `type="submit"` buttons. - **Testing:** Vitest or Jest; Supabase CLI / Testcontainers for ephemeral Postgres; `aws-sdk-client-mock` for AWS SDK; LocalStack for the thin E2E tier.

## Non-negotiable invariants

These are the properties the whole system exists to guarantee. Every change must preserve them; they are pass/fail acceptance gates (┬¦18), not aspirations.

1. **Tenant isolation.** Every tenant-scoped row carries `workspace_id NOT NULL`. Isolation is defense-in-depth: (a) app code filters by `workspace_id` in *every* query, (b) Postgres RLS for user-context (admin app) connections. **Critical caveat (┬¦3):** backend processing Lambdas connect with the Supabase **service role, which bypasses RLS** ŌĆö they MUST scope by `workspace_id` in code; RLS is not their safety net.

2. **Workspace is never client-supplied.** Derive it from the API key at ingest (┬¦7) and from the authorizer-injected claim on the admin API (┬¦12). Never trust a `workspace_id` in a request payload.

3. **Per-profile event ordering.** `MessageGroupId = profile_id` gives per-profile FIFO. Code must be **idempotent and order-convergent** so a `progress` arriving before `profile_created` still resolves correctly (upsert a stub). Don't unit-test "the queue delivered in order" ŌĆö that's SQS's job; test your convergence.

4. **No lost events.** `200` is returned only after SQS accepts (durable boundary). Forced failures retry, then go to DLQ ŌĆö never vanish.

5. **Idempotency.** `events.event_id` is producer-supplied and the dedupe key (`INSERT ... ON CONFLICT DO NOTHING`); `outbox.dedupe_key` and broadcast `(broadcast_id, profile_id)` prevent double-sends.

6. **Segment SQL is compiled, never interpolated.** The rule-AST ŌåÆ SQL compiler (┬¦8) whitelists fields/operators, emits parameterized SQL only, and **always** prepends `workspace_id = $ws`. This is security-critical and the single highest-value unit-test target.

7. **Sending is gated on verification.** The Dispatcher refuses to send for any workspace not `active`/verified (┬¦10). Every send passes suppression ŌåÆ frequency-cap ŌåÆ quiet-hours checks. **Guardrail source (v0.99.0):** the frequency cap + quiet-hours are read from `workspaces.settings` (`frequency_cap_per_days` int, 0=off; `quiet_hours {startHour,endHour}` UTC, null=off), configured in Workspace-settings ŌåÆ Sending guardrails; before v0.99.0 these two stages were inert (no send path stamped them). Suppression + soft-bounce are EMAIL-keyed and only apply to `medium==='email'` (a text send is never blocked by an email bounce). **Verification source (overrides ┬¦10A):** the gate (`canSend`) is fed from a VERIFIED row in `sending_domains` ŌĆö the per-domain model's source of truth ŌĆö derived in `services/dispatcher/src/dispatch.ts` (a verified domain also supplies the `no-reply@<domain>` From fallback). The legacy `workspaces.sending_identity` jsonb is only a fallback for old data; the per-domain onboarding flow never populates it, so gating on it alone silently refused fully-verified workspaces (regression test: `services/dispatcher/test/verified-domain-gate.integration.test.ts`).

## Roles (┬¦3A)

Four roles. `system-admin` (in `platform_admins`) is the **only cross-tenant role** ŌĆö the authorizer injects an `is_platform_admin` claim, RLS has a narrow exception for it, and every cross-tenant access is written to `admin_audit_log`. The other three (`owner` / `marketer` / `accounting`) are workspace-scoped via `workspace_users.role`; a user may hold different roles in different workspaces.

## Architecture flow

```
Producer --API key--> API GW REST (validate, usage plan, WAF)
  --> Ingest Lambda (resolve workspace from key, upsert profile) --> SQS FIFO (group=profile_id)
  --> Processor (idempotent, workspace-scoped): events -> profile_features -> re-eval segments -> membership diff + change_log
        -> segment entry drives automation enrollment (┬¦9B) and is available to broadcasts (┬¦9A)
  --> outbox row -> 2nd SQS queue --> Dispatcher: suppression -> freq-cap -> quiet-hours -> SES (workspace config set) -> messages_log
SES events -> SNS -> Feedback Lambda -> per-workspace suppression + reputation policing (auto-suspend offender)
```

Broadcasts (┬¦9A) and automations (┬¦9B) both emit sends through the **same** outbox ŌåÆ Dispatcher ŌåÆ SES pipeline. Automations are a **table-driven state machine** over `automation_enrollments`, swept by a scheduled Automation-runner Lambda (chosen over Step Functions: cheaper for long multi-day waits at this scale).

**Automation trigger kinds + enrollment (┬¦9B, phase 3):** a automation's `definition.startNode` is its single **trigger node** ŌĆö one of FOUR `kind`s (`services/automation-runner/src/dsl.ts`):
- **`segment_entry`** ŌĆö the trigger segment lives on the **automation ROW** (`automations.trigger_segment_id` + `trigger_on` enter|exit), NOT on the node. Entering/leaving that segment drives `enrollFromSegmentChange`.
- **`event`** ŌĆö the node carries `eventType` (REQUIRED) + an optional **payload filter** (`filter`, a `payload.*`-namespace `AstNode`); BOTH live in the **definition JSON** (no migration). An ingested event of that type drives `enrollFromEvent`. The filter is a **pure, closed-grammar, in-memory** eval over the event payload (`evaluateEventPayloadFilter` in `core.ts` ŌĆö reuses the `@cdp/segments` operator whitelist via `resolveOperator`; a non-whitelisted operator or a non-`payload.*` field THROWS ŌĆö never interpolated). **A dotted payload key is resolved DEEP and FORGIVING (`webinar_data?.id` semantics, v0.49.0):** `payload.webinar_data.id` descends nested objects/arrays via the shared `resolveEventPath` (array indices supported, e.g. `items.0.sku`) and yields `undefined` for ANY missing segment ŌĆö so `webinar_data.id exists` is `false` (NEVER throws) when `webinar_data` is absent, and `true` only when the nested leaf is present. A flat key (no dot) behaves exactly like the prior `payload[key]` lookup.
- **`profile`** (v0.43.0) ŌĆö enrolls when a PROFILE is **created or updated**. The node carries `profileChange: 'created' | 'updated' | 'any'` (in the definition JSON, no migration; default `'any'` ŌĆö `validateAutomationDefinition` rejects any other value). A profile create/update drives `enrollFromProfileChange(deps, { workspace_id, profile_id, change })` (`enroll.ts`, parsed by `parseProfileEnrollmentTrigger` in `core.ts`): a `created` change matches `profileChange Ōłł {created, any}`, an `updated` change matches `{updated, any}`. NO event payload is persisted on `state` (an `{{event.*}}` expression downstream resolves safe-empty); the profile's own data is read via `customer.*`. The trigger card summary reads "On profile created" / "On profile updated" / "On profile created or updated" (`nodeSummary` in `web/src/automations/mutate.ts`); the editor exposes a `trigger-profile-change` Select (`NodeEditor.tsx` TriggerEditor) carried by `read/writeTriggerConfig` (`node-config.ts`).
- **`manual`** ŌĆö no auto-source; enrolled by the API (`POST /automations/:id/enroll`).

**Where enrollment fires (same workspace-scoped tx as segment re-eval, no nested BEGIN/COMMIT):** right after the event row + feature recompute + realtime segment re-eval, on the SAME tx client. **PROD:** `services/processor` `runPlanInWorkspaceTx ŌåÆ runSegmentReevalInTx` (the `SegmentReeval` marker now also carries `eventType`/`eventPayload`/`eventId`) invokes `enrollFromSegmentChange` per entered/exited delta AND `enrollFromEvent`. **DEV mirror:** `services/local-api` `recomputeFeaturesAndSegments` (called by `sendProfileEvent` + the merge path) fires the same two; `sendProfileEvent` also calls `enrollFromEvent` directly with the just-recorded event. **Profile-trigger** (`enrollFromProfileChange`) fires from the profile mutation handlers in `services/local-api/src/handlers.ts`: `createProfile` (`change='created'`), `updateProfile`/`PATCH /profiles/:id` (`change='updated'`), and `importProfilesCsv` (`change='created'` per imported/created row) ŌĆö all on the request pool's own workspace-scoped tx (`enrollDepsOnPool`), idempotent via ON CONFLICT 'once'. (Before phase 3, `enrollFromSegmentChange` was exported but NOT invoked by any live path ŌĆö phase 3 wired it into the live flow.) **Manual/API:** `POST /automations/:id/enroll` (`enrollIntoAutomation`, capability `manage_content`) takes **exactly one** of `profile_id` | `segment_id`, scopes the automation/profile/segment to `ctx.workspaceId` (a cross-workspace id ŌåÆ 404, inv.2), and enrolls the profile or a **point-in-time SEGMENT SNAPSHOT** (current members via `buildResolveAudience` ŌĆö both dynamic + manual sources; later segment changes do NOT retroactively enroll). `workspace_id` is from the authorizer ctx ONLY ŌĆö a body `workspace_id` is ignored.

**Re-enrollment policy = `'once'` for ALL kinds (the documented default).** Enforced structurally by `UNIQUE(automation_id, profile_id)` + `INSERT ŌĆ” ON CONFLICT DO NOTHING` (`buildEnrollmentInsert`) and decided in code by `decideReenrollment(hasExisting, 'once')` (`DEFAULT_REENROLLMENT_POLICY` in `core.ts`). All kinds funnel through `buildEnrollmentInsert`, so a replayed event / repeated segment entry / re-run manual enroll inserts at most one enrollment. `'always'` (re-entry) stays a forward-compat enum value but is NOT enabled (no migration; the policy is a code default ŌĆö if ever persisted it would be a single `automations.reenrollment_policy` column). No sends are emitted by enrollment (enrollment writes `automation_enrollments` only; the runner sweep later drives sends through the Dispatcher).

## How to build (until tooling exists, this is the contract)

- **Build tenancy first** (workspaces, `workspace_id` everywhere, RLS, authorizer) before any feature, so nothing is retrofitted.
- **Work phase by phase (┬¦17)**; a phase is done only when its ┬¦18 acceptance criteria pass as tests in CI.
- **Test-first (┬¦16A):** write the phase's ┬¦18 criteria as failing tests, then implement to green.
  - **Don't mock Postgres** in the integration tier ŌĆö ordering, idempotency, and isolation bugs live in the SQL and RLS. Use local Supabase (Supabase CLI) or Testcontainers.
  - **Do mock SES** ŌĆö assert `SendEmail` is called with the right Configuration Set, only after suppression/cap checks; never send real mail.
- Local dev: SAM local / serverless-offline + LocalStack (SQS/S3/SNS/API GW) + Supabase CLI. Seed multiple workspaces with overlapping `external_id`s/emails to test isolation.
