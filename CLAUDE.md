## What is being built

A **serverless, multi-tenant marketing CDP** on AWS: companies get isolated workspaces, ingest behavioral events over HTTP, maintain unified customer profiles with rolling aggregates, define segments, and reach people via broadcasts (one-off blasts) and campaigns (multi-step journeys) through Amazon SES. An admin SPA provides segment/campaign builders, a WYSIWYG email editor, dashboards, and per-workspace cost views.

## Tech stack (all decisions are locked ŌĆö see ┬¦0)

- **Backend:** TypeScript on Node.js 20+, AWS Lambda (real Node runtime, full npm incl. `sharp`).
- **HTTP entry:** API Gateway **REST API** (not HTTP API ŌĆö chosen for request validation, per-workspace API keys + usage plans, native WAF).
- **Database:** Supabase (managed PostgreSQL), pooled connection from Lambda. System of record.
- **Queue:** SQS **FIFO** (+ DLQ), `MessageGroupId = profile_id`.
- **Email:** Amazon SES; each workspace sends from its own verified domain.
- **Auth:** Supabase Auth; validated at the gateway by a **Lambda authorizer** (REST API has no native JWT authorizer).
- **Email editor:** a CUSTOM in-house designer (`web/src/email-designer/`, ported from the owner's nomentor builder; Preact + @preact/signals) ŌĆö this deliberately OVERRIDES the spec's GrapesJS choice. Templates are a LIBRARY; attaching one to a broadcast/campaign CLONES it (`kind='copy'`, `source_template_id`) into an independently editable working copy.
- **IaC:** AWS CDK (TypeScript).
- **Frontend:** Preact/React SPA, Vite, static on S3 + CloudFront. **Server-calling buttons auto-lock:** the shared kit `Button` (`web/src/ui/kit.tsx`) shows a spinner + disables itself whenever its `onClick` returns a Promise ŌĆö so wire every server-calling button so its handler RETURNS the promise (`onClick={save}` / `onClick={() => save(id)}`, never `() => void save()`), and pass `loading` explicitly for `type="submit"` buttons. - **Testing:** Vitest or Jest; Supabase CLI / Testcontainers for ephemeral Postgres; `aws-sdk-client-mock` for AWS SDK; LocalStack for the thin E2E tier.

## Non-negotiable invariants

These are the properties the whole system exists to guarantee. Every change must preserve them; they are pass/fail acceptance gates (┬¦18), not aspirations.

1. **Tenant isolation.** Every tenant-scoped row carries `workspace_id NOT NULL`. Isolation is defense-in-depth: (a) app code filters by `workspace_id` in *every* query, (b) Postgres RLS for user-context (admin app) connections. **Critical caveat (┬¦3):** backend processing Lambdas connect with the Supabase **service role, which bypasses RLS** ŌĆö they MUST scope by `workspace_id` in code; RLS is not their safety net.

2. **Workspace is never client-supplied.** Derive it from the API key at ingest (┬¦7) and from the authorizer-injected claim on the admin API (┬¦12). Never trust a `workspace_id` in a request payload.

3. **Per-profile event ordering.** `MessageGroupId = profile_id` gives per-profile FIFO. Code must be **idempotent and order-convergent** so a `progress` arriving before `profile_created` still resolves correctly (upsert a stub). Don't unit-test "the queue delivered in order" ŌĆö that's SQS's job; test your convergence.

4. **No lost events.** `200` is returned only after SQS accepts (durable boundary). Forced failures retry, then go to DLQ ŌĆö never vanish.

5. **Idempotency.** `events.event_id` is producer-supplied and the dedupe key (`INSERT ... ON CONFLICT DO NOTHING`); `outbox.dedupe_key` and broadcast `(broadcast_id, profile_id)` prevent double-sends.

6. **Segment SQL is compiled, never interpolated.** The rule-AST ŌåÆ SQL compiler (┬¦8) whitelists fields/operators, emits parameterized SQL only, and **always** prepends `workspace_id = $ws`. This is security-critical and the single highest-value unit-test target.

7. **Sending is gated on verification.** The Dispatcher refuses to send for any workspace not `active`/verified (┬¦10). Every send passes suppression ŌåÆ frequency-cap ŌåÆ quiet-hours checks. **Verification source (overrides ┬¦10A):** the gate (`canSend`) is fed from a VERIFIED row in `sending_domains` ŌĆö the per-domain model's source of truth ŌĆö derived in `services/dispatcher/src/dispatch.ts` (a verified domain also supplies the `no-reply@<domain>` From fallback). The legacy `workspaces.sending_identity` jsonb is only a fallback for old data; the per-domain onboarding flow never populates it, so gating on it alone silently refused fully-verified workspaces (regression test: `services/dispatcher/test/verified-domain-gate.integration.test.ts`).

## Roles (┬¦3A)

Four roles. `system-admin` (in `platform_admins`) is the **only cross-tenant role** ŌĆö the authorizer injects an `is_platform_admin` claim, RLS has a narrow exception for it, and every cross-tenant access is written to `admin_audit_log`. The other three (`owner` / `marketer` / `accounting`) are workspace-scoped via `workspace_users.role`; a user may hold different roles in different workspaces.

## Architecture flow

```
Producer --API key--> API GW REST (validate, usage plan, WAF)
  --> Ingest Lambda (resolve workspace from key, upsert profile) --> SQS FIFO (group=profile_id)
  --> Processor (idempotent, workspace-scoped): events -> profile_features -> re-eval segments -> membership diff + change_log
        -> segment entry drives campaign enrollment (┬¦9B) and is available to broadcasts (┬¦9A)
  --> outbox row -> 2nd SQS queue --> Dispatcher: suppression -> freq-cap -> quiet-hours -> SES (workspace config set) -> messages_log
SES events -> SNS -> Feedback Lambda -> per-workspace suppression + reputation policing (auto-suspend offender)
```

Broadcasts (┬¦9A) and campaigns (┬¦9B) both emit sends through the **same** outbox ŌåÆ Dispatcher ŌåÆ SES pipeline. Campaigns are a **table-driven state machine** over `campaign_enrollments`, swept by a scheduled Campaign-runner Lambda (chosen over Step Functions: cheaper for long multi-day waits at this scale).

**Campaign trigger kinds + enrollment (┬¦9B, phase 3):** a campaign's `definition.startNode` is its single **trigger node** ŌĆö one of FOUR `kind`s (`services/campaign-runner/src/dsl.ts`):
- **`segment_entry`** ŌĆö the trigger segment lives on the **campaign ROW** (`campaigns.trigger_segment_id` + `trigger_on` enter|exit), NOT on the node. Entering/leaving that segment drives `enrollFromSegmentChange`.
- **`event`** ŌĆö the node carries `eventType` (REQUIRED) + an optional **payload filter** (`filter`, a `payload.*`-namespace `AstNode`); BOTH live in the **definition JSON** (no migration). An ingested event of that type drives `enrollFromEvent`. The filter is a **pure, closed-grammar, in-memory** eval over the event payload (`evaluateEventPayloadFilter` in `core.ts` ŌĆö reuses the `@cdp/segments` operator whitelist via `resolveOperator`; a non-whitelisted operator or a non-`payload.*` field THROWS ŌĆö never interpolated). **A dotted payload key is resolved DEEP and FORGIVING (`webinar_data?.id` semantics, v0.49.0):** `payload.webinar_data.id` descends nested objects/arrays via the shared `resolveEventPath` (array indices supported, e.g. `items.0.sku`) and yields `undefined` for ANY missing segment ŌĆö so `webinar_data.id exists` is `false` (NEVER throws) when `webinar_data` is absent, and `true` only when the nested leaf is present. A flat key (no dot) behaves exactly like the prior `payload[key]` lookup.
- **`profile`** (v0.43.0) ŌĆö enrolls when a PROFILE is **created or updated**. The node carries `profileChange: 'created' | 'updated' | 'any'` (in the definition JSON, no migration; default `'any'` ŌĆö `validateCampaignDefinition` rejects any other value). A profile create/update drives `enrollFromProfileChange(deps, { workspace_id, profile_id, change })` (`enroll.ts`, parsed by `parseProfileEnrollmentTrigger` in `core.ts`): a `created` change matches `profileChange Ōłł {created, any}`, an `updated` change matches `{updated, any}`. NO event payload is persisted on `state` (an `{{event.*}}` expression downstream resolves safe-empty); the profile's own data is read via `customer.*`. The trigger card summary reads "On profile created" / "On profile updated" / "On profile created or updated" (`nodeSummary` in `web/src/campaigns/mutate.ts`); the editor exposes a `trigger-profile-change` Select (`NodeEditor.tsx` TriggerEditor) carried by `read/writeTriggerConfig` (`node-config.ts`).
- **`manual`** ŌĆö no auto-source; enrolled by the API (`POST /campaigns/:id/enroll`).

**Where enrollment fires (same workspace-scoped tx as segment re-eval, no nested BEGIN/COMMIT):** right after the event row + feature recompute + realtime segment re-eval, on the SAME tx client. **PROD:** `services/processor` `runPlanInWorkspaceTx ŌåÆ runSegmentReevalInTx` (the `SegmentReeval` marker now also carries `eventType`/`eventPayload`/`eventId`) invokes `enrollFromSegmentChange` per entered/exited delta AND `enrollFromEvent`. **DEV mirror:** `services/local-api` `recomputeFeaturesAndSegments` (called by `sendProfileEvent` + the merge path) fires the same two; `sendProfileEvent` also calls `enrollFromEvent` directly with the just-recorded event. **Profile-trigger** (`enrollFromProfileChange`) fires from the profile mutation handlers in `services/local-api/src/handlers.ts`: `createProfile` (`change='created'`), `updateProfile`/`PATCH /profiles/:id` (`change='updated'`), and `importProfilesCsv` (`change='created'` per imported/created row) ŌĆö all on the request pool's own workspace-scoped tx (`enrollDepsOnPool`), idempotent via ON CONFLICT 'once'. (Before phase 3, `enrollFromSegmentChange` was exported but NOT invoked by any live path ŌĆö phase 3 wired it into the live flow.) **Manual/API:** `POST /campaigns/:id/enroll` (`enrollIntoCampaign`, capability `manage_content`) takes **exactly one** of `profile_id` | `segment_id`, scopes the campaign/profile/segment to `ctx.workspaceId` (a cross-workspace id ŌåÆ 404, inv.2), and enrolls the profile or a **point-in-time SEGMENT SNAPSHOT** (current members via `buildResolveAudience` ŌĆö both dynamic + manual sources; later segment changes do NOT retroactively enroll). `workspace_id` is from the authorizer ctx ONLY ŌĆö a body `workspace_id` is ignored.

**Re-enrollment policy = `'once'` for ALL kinds (the documented default).** Enforced structurally by `UNIQUE(campaign_id, profile_id)` + `INSERT ŌĆ” ON CONFLICT DO NOTHING` (`buildEnrollmentInsert`) and decided in code by `decideReenrollment(hasExisting, 'once')` (`DEFAULT_REENROLLMENT_POLICY` in `core.ts`). All kinds funnel through `buildEnrollmentInsert`, so a replayed event / repeated segment entry / re-run manual enroll inserts at most one enrollment. `'always'` (re-entry) stays a forward-compat enum value but is NOT enabled (no migration; the policy is a code default ŌĆö if ever persisted it would be a single `campaigns.reenrollment_policy` column). No sends are emitted by enrollment (enrollment writes `campaign_enrollments` only; the runner sweep later drives sends through the Dispatcher).

### Status of scaffolding

**Company ŌåÆ Workspace hierarchy (extends spec ┬¦6):** a `companies` table groups workspaces ŌĆö every `workspace` has `company_id NOT NULL` (migration `0012`). Tenant **isolation is unchanged**: it stays at the workspace level (every tenant row carries `workspace_id`; RLS still keys on `workspace_id`). A company is purely the organizational parent so a platform admin can pick a company ŌåÆ a workspace within it (sidebar `CompanyWorkspacePicker`, fed by `GET /admin/companies`). Workspaces created without a `company_id` (chiefly in integration tests) are auto-assigned to a shared `Unassigned` company by a BEFORE-INSERT trigger (migration `0013`); the product's own creation paths (registration, `POST /workspaces`) always supply a real `company_id`, so `Unassigned` never appears from normal use. **A user belongs to ONE company:** access is per-workspace (`workspace_users`), but `addMember` rejects adding a user to a workspace whose company differs from one they're already in (409) ŌĆö so a user's memberships always stay within a single company (which may own several workspaces).

**Registration creates a COMPANY only ŌĆö never a workspace (the owner creates the first one manually).** `POST /auth/register` (`registerOwner`, local-api `session.ts`) inserts the owner `users` row + a `companies` row tagged with `owner_user_id` (migration `0032`) ŌĆö NO workspace, NO `workspace_users`. The minted token is logged-in but workspace-less, so the response carries `needs_workspace: true` + the `company`. Because the production `authorize()` core hard-denies a non-platform-admin token with no active workspace, the workspace-less owner can't reach any capability-gated route (not even `/me`); so creating the first workspace is a **session route** `POST /workspace/bootstrap` (`createFirstWorkspace`) ŌĆö like `/workspace/switch`, it authenticates the token directly, resolves the company server-side from `owner_user_id = <token sub>` (never client-supplied, inv.2), inserts the workspace + owner `workspace_users`, and **re-mints the token** with the new active workspace. `devLogin` mirrors this: a workspace-less owner who owns a company logs in 200 with `needs_workspace` (instead of the old 403); anyone else with no membership is still 403. SPA: `App.tsx` routes a logged-in non-admin with `!workspaceId` to the `CreateFirstWorkspace` screen (session `needsWorkspace`); `restoreSession` skips `/me` in that state; `bootstrapWorkspace()` enters the shell. Additional workspaces (for an owner already in one) are still created via the capability-gated `POST /workspaces` in Company settings.

**Identity key (overrides spec ┬¦6/┬¦7):** a profile's per-workspace identity/merge key is **`email`**, NOT `external_id`. Events arrive from many source systems; email is the only identifier that stitches a person's events together, so ingestion requires `email`, profiles are `UNIQUE(workspace_id, email)` (migration `0010`), and the ingest/processor upserts + the manual `POST /profiles` all merge on `(workspace_id, email)`. `external_id` is optional metadata (the spec's original dedupe-by-external_id design is superseded).

**Sending domains are a LIST with a per-domain setup screen (extends ┬¦10):** a workspace can have several sending domains ŌĆö `sending_domains` table (migration `0021`, workspace-scoped + RLS), each with a `verified` flag. Verification uses **real SES**: on first open each domain is provisioned as an SES email identity (`SesEmailClient.createDomainIdentity` ŌåÆ 3 DKIM tokens, stored in `sending_domains.ses_identity` + `dkim_tokens`, migration `0023`); the check calls `getIdentityVerificationAttributes` and verifies ONLY when `DkimStatus === 'SUCCESS'` ŌĆö no manual flip. **The DKIM CNAME target host is read FROM SES, never constructed** (multi-region: each company picks its own SES region): SES returns `DkimAttributes.SigningHostedZone` (region-specific, e.g. `dkim.il-central-1.amazonses.com`), persisted in `sending_domains.signing_hosted_zone` (migration `0029`) and used verbatim as `<token>.<signing_hosted_zone>` (`dkimCnameHost` in `handlers.ts`; pre-`0029` rows backfill on next load). **MOVE / DUPLICATE a node + its branch ŌĆö exclusive-subtree placement (v0.39.0, extends ┬¦9B):** the per-node `ActionMenu` (non-trigger, non-exit) gains **Move toŌĆ”** (`node-move`) and **DuplicateŌĆ”** (`node-duplicate`) alongside Edit/Delete.
