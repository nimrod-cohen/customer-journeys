## What is being built

A **serverless, multi-tenant marketing CDP** on AWS: companies get isolated workspaces, ingest behavioral events over HTTP, maintain unified customer profiles with rolling aggregates, define segments, and reach people via broadcasts (one-off blasts) and automations (multi-step journeys) through Amazon SES. An admin SPA provides segment/automation builders, a WYSIWYG email editor, dashboards, and per-workspace cost views.

## Tech stack (all decisions are locked — see §0)

- **Backend:** TypeScript on Node.js 20+, AWS Lambda (real Node runtime, full npm incl. `sharp`).
- **HTTP entry:** API Gateway **REST API** (not HTTP API — chosen for request validation, per-workspace API keys + usage plans, native WAF).
- **Database:** Supabase (managed PostgreSQL), pooled connection from Lambda. System of record.
- **Queue:** SQS **FIFO** (+ DLQ), `MessageGroupId = profile_id`.
- **Email:** Amazon SES; each workspace sends from its own verified domain.
- **Auth:** Supabase Auth; validated at the gateway by a **Lambda authorizer** (REST API has no native JWT authorizer).
- **Email editor:** a CUSTOM in-house designer (`web/src/email-designer/`, ported from the owner's nomentor builder; Preact + @preact/signals) — this deliberately OVERRIDES the spec's GrapesJS choice. The surviving invariant is unchanged: the editor **emits MJML, never hand-rolled HTML** (design JSON → `mjml-serializer.ts` → server `compileMjml`, strict). Templates are a LIBRARY; attaching one to a broadcast/automation CLONES it (`kind='copy'`, `source_template_id`) into an independently editable working copy.
- **IaC:** AWS CDK (TypeScript).
- **Frontend:** Preact/React SPA, Vite, static on S3 + CloudFront. **Server-calling buttons auto-lock:** the shared kit `Button` (`web/src/ui/kit.tsx`) shows a spinner + disables itself whenever its `onClick` returns a Promise — so wire every server-calling button so its handler RETURNS the promise (`onClick={save}` / `onClick={() => save(id)}`, never `() => void save()`), and pass `loading` explicitly for `type="submit"` buttons. This is a standing rule (no double-submits, clear in-flight state).
- **Testing:** Vitest or Jest; Supabase CLI / Testcontainers for ephemeral Postgres; `aws-sdk-client-mock` for AWS SDK; LocalStack for the thin E2E tier.

## Non-negotiable invariants

These are the properties the whole system exists to guarantee. Every change must preserve them; they are pass/fail acceptance gates (§18), not aspirations.

1. **Tenant isolation.** Every tenant-scoped row carries `workspace_id NOT NULL`. Isolation is defense-in-depth: (a) app code filters by `workspace_id` in *every* query, (b) Postgres RLS for user-context (admin app) connections. **Critical caveat (§3):** backend processing Lambdas connect with the Supabase **service role, which bypasses RLS** — they MUST scope by `workspace_id` in code; RLS is not their safety net.

2. **Workspace is never client-supplied.** Derive it from the API key at ingest (§7) and from the authorizer-injected claim on the admin API (§12). Never trust a `workspace_id` in a request payload.

3. **Per-profile event ordering.** `MessageGroupId = profile_id` gives per-profile FIFO. Code must be **idempotent and order-convergent** so a `progress` arriving before `profile_created` still resolves correctly (upsert a stub). Don't unit-test "the queue delivered in order" — that's SQS's job; test your convergence.

4. **No lost events.** `200` is returned only after SQS accepts (durable boundary). Forced failures retry, then go to DLQ — never vanish.

5. **Idempotency.** `events.event_id` is producer-supplied and the dedupe key (`INSERT ... ON CONFLICT DO NOTHING`); `outbox.dedupe_key` and broadcast `(broadcast_id, profile_id)` prevent double-sends.

6. **Segment SQL is compiled, never interpolated.** The rule-AST → SQL compiler (§8) whitelists fields/operators, emits parameterized SQL only, and **always** prepends `workspace_id = $ws`. This is security-critical and the single highest-value unit-test target.

7. **Sending is gated on verification.** The Dispatcher refuses to send for any workspace not `active`/verified (§10). Every send passes suppression → frequency-cap → quiet-hours checks. **Guardrail source (v0.99.0):** the frequency cap + quiet-hours are read from `workspaces.settings` (`frequency_cap_per_days` int, 0=off; `quiet_hours {startHour,endHour}` UTC, null=off), configured in Workspace-settings → Sending guardrails; before v0.99.0 these two stages were inert (no send path stamped them). Suppression + soft-bounce are EMAIL-keyed and only apply to `medium==='email'` (a text send is never blocked by an email bounce). **Verification source (overrides §10A):** the gate (`canSend`) is fed from a VERIFIED row in `sending_domains` — the per-domain model's source of truth — derived in `services/dispatcher/src/dispatch.ts` (a verified domain also supplies the `no-reply@<domain>` From fallback). The legacy `workspaces.sending_identity` jsonb is only a fallback for old data; the per-domain onboarding flow never populates it, so gating on it alone silently refused fully-verified workspaces (regression test: `services/dispatcher/test/verified-domain-gate.integration.test.ts`).

## Roles (§3A)

Four roles. `system-admin` (in `platform_admins`) is the **only cross-tenant role** — the authorizer injects an `is_platform_admin` claim, RLS has a narrow exception for it, and every cross-tenant access is written to `admin_audit_log`. The other three (`owner` / `marketer` / `accounting`) are workspace-scoped via `workspace_users.role`; a user may hold different roles in different workspaces.

## Architecture flow

```
Producer --API key--> API GW REST (validate, usage plan, WAF)
  --> Ingest Lambda (resolve workspace from key, upsert profile) --> SQS FIFO (group=profile_id)
  --> Processor (idempotent, workspace-scoped): events -> profile_features -> re-eval segments -> membership diff + change_log
        -> segment entry drives automation enrollment (§9B) and is available to broadcasts (§9A)
  --> outbox row -> 2nd SQS queue --> Dispatcher: suppression -> freq-cap -> quiet-hours -> SES (workspace config set) -> messages_log
SES events -> SNS -> Feedback Lambda -> per-workspace suppression + reputation policing (auto-suspend offender)
```

Broadcasts (§9A) and automations (§9B) both emit sends through the **same** outbox → Dispatcher → SES pipeline. Automations are a **table-driven state machine** over `automation_enrollments`, swept by a scheduled Automation-runner Lambda (chosen over Step Functions: cheaper for long multi-day waits at this scale).

**Automation trigger kinds + enrollment (§9B, phase 3):** a automation's `definition.startNode` is its single **trigger node** — one of FOUR `kind`s (`services/automation-runner/src/dsl.ts`):
- **`segment_entry`** — the trigger segment lives on the **automation ROW** (`automations.trigger_segment_id` + `trigger_on` enter|exit), NOT on the node. Entering/leaving that segment drives `enrollFromSegmentChange`.
- **`event`** — the node carries `eventType` (REQUIRED) + an optional **payload filter** (`filter`, a `payload.*`-namespace `AstNode`); BOTH live in the **definition JSON** (no migration). An ingested event of that type drives `enrollFromEvent`. The filter is a **pure, closed-grammar, in-memory** eval over the event payload (`evaluateEventPayloadFilter` in `core.ts` — reuses the `@cdp/segments` operator whitelist via `resolveOperator`; a non-whitelisted operator or a non-`payload.*` field THROWS — never interpolated). **A dotted payload key is resolved DEEP and FORGIVING (`webinar_data?.id` semantics, v0.49.0):** `payload.webinar_data.id` descends nested objects/arrays via the shared `resolveEventPath` (array indices supported, e.g. `items.0.sku`) and yields `undefined` for ANY missing segment — so `webinar_data.id exists` is `false` (NEVER throws) when `webinar_data` is absent, and `true` only when the nested leaf is present. A flat key (no dot) behaves exactly like the prior `payload[key]` lookup.
- **`profile`** (v0.43.0) — enrolls when a PROFILE is **created or updated**. The node carries `profileChange: 'created' | 'updated' | 'any'` (in the definition JSON, no migration; default `'any'` — `validateAutomationDefinition` rejects any other value). A profile create/update drives `enrollFromProfileChange(deps, { workspace_id, profile_id, change })` (`enroll.ts`, parsed by `parseProfileEnrollmentTrigger` in `core.ts`): a `created` change matches `profileChange ∈ {created, any}`, an `updated` change matches `{updated, any}`. NO event payload is persisted on `state` (an `{{event.*}}` expression downstream resolves safe-empty); the profile's own data is read via `customer.*`. The trigger card summary reads "On profile created" / "On profile updated" / "On profile created or updated" (`nodeSummary` in `web/src/automations/mutate.ts`); the editor exposes a `trigger-profile-change` Select (`NodeEditor.tsx` TriggerEditor) carried by `read/writeTriggerConfig` (`node-config.ts`).
- **`manual`** — no auto-source; enrolled by the API (`POST /automations/:id/enroll`).

**Where enrollment fires (same workspace-scoped tx as segment re-eval, no nested BEGIN/COMMIT):** right after the event row + feature recompute + realtime segment re-eval, on the SAME tx client. **PROD:** `services/processor` `runPlanInWorkspaceTx → runSegmentReevalInTx` (the `SegmentReeval` marker now also carries `eventType`/`eventPayload`/`eventId`) invokes `enrollFromSegmentChange` per entered/exited delta AND `enrollFromEvent`. **DEV mirror:** `services/local-api` `recomputeFeaturesAndSegments` (called by `sendProfileEvent` + the merge path) fires the same two; `sendProfileEvent` also calls `enrollFromEvent` directly with the just-recorded event. **Profile-trigger** (`enrollFromProfileChange`) fires from the profile mutation handlers in `services/local-api/src/handlers.ts`: `createProfile` (`change='created'`), `updateProfile`/`PATCH /profiles/:id` (`change='updated'`), and `importProfilesCsv` (`change='created'` per imported/created row) — all on the request pool's own workspace-scoped tx (`enrollDepsOnPool`), idempotent via ON CONFLICT 'once'. (Before phase 3, `enrollFromSegmentChange` was exported but NOT invoked by any live path — phase 3 wired it into the live flow.) **Manual/API:** `POST /automations/:id/enroll` (`enrollIntoAutomation`, capability `manage_content`) takes **exactly one** of `profile_id` | `segment_id`, scopes the automation/profile/segment to `ctx.workspaceId` (a cross-workspace id → 404, inv.2), and enrolls the profile or a **point-in-time SEGMENT SNAPSHOT** (current members via `buildResolveAudience` — both dynamic + manual sources; later segment changes do NOT retroactively enroll). `workspace_id` is from the authorizer ctx ONLY — a body `workspace_id` is ignored.

**Re-enrollment policy = `'once'` for ALL kinds (the documented default).** Enforced structurally by `UNIQUE(automation_id, profile_id)` + `INSERT … ON CONFLICT DO NOTHING` (`buildEnrollmentInsert`) and decided in code by `decideReenrollment(hasExisting, 'once')` (`DEFAULT_REENROLLMENT_POLICY` in `core.ts`). All kinds funnel through `buildEnrollmentInsert`, so a replayed event / repeated segment entry / re-run manual enroll inserts at most one enrollment. `'always'` (re-entry) stays a forward-compat enum value but is NOT enabled (no migration; the policy is a code default — if ever persisted it would be a single `automations.reenrollment_policy` column). No sends are emitted by enrollment (enrollment writes `automation_enrollments` only; the runner sweep later drives sends through the Dispatcher).

## How to build (until tooling exists, this is the contract)

- **Build tenancy first** (workspaces, `workspace_id` everywhere, RLS, authorizer) before any feature, so nothing is retrofitted.
- **Work phase by phase (§17)**; a phase is done only when its §18 acceptance criteria pass as tests in CI.
- **Test-first (§16A):** write the phase's §18 criteria as failing tests, then implement to green.
  - **Don't mock Postgres** in the integration tier — ordering, idempotency, and isolation bugs live in the SQL and RLS. Use local Supabase (Supabase CLI) or Testcontainers.
  - **Do mock SES** — assert `SendEmail` is called with the right Configuration Set, only after suppression/cap checks; never send real mail.
- Local dev: SAM local / serverless-offline + LocalStack (SQS/S3/SNS/API GW) + Supabase CLI. Seed multiple workspaces with overlapping `external_id`s/emails to test isolation.

**Registration creates a COMPANY only — never a workspace (the owner creates the first one manually).** `POST /auth/register` (`registerOwner`, local-api `session.ts`) inserts the owner `users` row + a `companies` row tagged with `owner_user_id` (migration `0032`) — NO workspace, NO `workspace_users`. **Identity key (overrides spec §6/§7):** a profile's per-workspace identity/merge key is **`email`**, NOT `external_id`. Events arrive from many source systems; email is the only identifier that stitches a person's events together, so ingestion requires `email`, profiles are `UNIQUE(workspace_id, email)` (migration `0010`), and the ingest/processor upserts + the manual `POST /profiles` all merge on `(workspace_id, email)`. `external_id` is optional metadata (the spec's original dedupe-by-external_id design is superseded).

**PHONE is a SECOND core identity field (v0.112.0, migration `0062`, extends the identity model):** a profile can be identified by **email AND/OR phone** — both are CORE, RESERVED fields (not dynamic attributes); each alone is optional but **at least one is required**. Phones are stored **normalized to E.164** so `+972541111111` == `054-1111111` (IL).
- **Schema (`0062`):** `profiles.phone text` (E.164, nullable), partial `UNIQUE(workspace_id, phone) WHERE phone IS NOT NULL`, a `workspace_id`-leading index, and `CHECK (email IS NOT NULL OR phone IS NOT NULL)` (`profiles_identity_present`). email stays nullable (a profile may be phone-only). **NOTE for tests:** a profile insert now MUST include an email OR phone (many fixtures set `email = external_id` to satisfy this).
- **Reserved fields (`@cdp/shared` customer.ts):** `RESERVED_CUSTOMER_FIELDS` gained `phone`. The `customer.*` resolver + `customerMerge` treat `email`, `phone`, and their `attributes.email`/`attributes.phone` spellings as the CORE columns (`CORE_ATTRIBUTE_ALIASES`) — so `{{customer.phone}}`, `{{customer.attributes.phone}}`, and a segment rule `customer.phone` (compiler `SCALAR_PROFILE_FIELDS` gained `phone: 'p.phone'`) all hit the column. **`customerMerge` FALLS BACK to `attributes.phone`/`attributes.email` when the column is empty** (so `{{customer.phone}}` still resolves for pre-backfill profiles that keep phone in attributes). No column (jsonb settings, no migration).
- **Identity resolution (`services/local-api/src/identity.ts` `resolveIdentity`, pure):** validate email (format) + normalize phone; **a bad phone with a valid email is DROPPED (record kept); a phone-only bad number is REJECTED (400)**; ≥1 key required. **MOVE / DUPLICATE a node + its branch — exclusive-subtree placement (v0.39.0, extends §9B):** the per-node `ActionMenu` (non-trigger, non-exit) gains **Move to…** (`node-move`) and **Duplicate…** (`node-duplicate`) alongside Edit/Delete. Both relocate/copy a node together with its **EXCLUSIVE SUBTREE** — pure ops in `web/src/automations/mutate.ts`, unit-tested first, re-validated locally AND server-side. **The algorithm (generalizes `falseArmExclusive`'s reachability):** for a root R, `S = fromR \ reachableWithoutR` (BFS from R, minus BFS from start treating R as absent), always including R; the **continuation** `C` = the single boundary target (a node that is an edge-target of some node in S but is NOT in S) — for a well-formed branch this is ONE node (the rejoin/join) or EMPTY (a terminal branch ending only in exits); **2+ distinct boundary targets ⇒ throw** `MutationError('This branch can't be moved as a unit.')`. NOTE a consequence of the tree shape: a node on the sole trunk pulls everything below it (incl. the exit) into S — a clean, single-node continuation only exists for a node on a DIAMOND ARM (the join is reachable via the sibling arm). **`moveSubtree(model, rootId, destEdge)`** guards (rootId≠start; destEdge.from/to ∉ S — else "Choose a spot outside the branch you're moving."; destEdge.to===rootId → no-op return), then **DETACHes** (R's parents → C via `repointParents`; dropped if C undefined → orphan caught by validate) and **REATTACHes** at A→B (A's slot→rootId; every boundary edge in S→C is re-pointed to B). **`duplicateSubtree(model, rootId, destEdge)`** clones every node in S with FRESH ids (`freshNodeId` per type via `paletteTypeOf`, an old→new id map), remaps INTERNAL edges to clones + BOUNDARY edges (→C) to B, and splices A→cloneRoot — originals untouched, fresh ids can't cycle. **MOVE vs DUPLICATE:** move re-points the originals (the source closes up around the gap, the branch relocates); duplicate leaves the originals intact and inserts an independent fresh-id copy. Both end with **LOCAL guards** `assertWellFormed` (`hasReachableExit` + no dangling edge + a new `hasCycle` DFS — down-only must hold; NO runner import in the web bundle) → throw on failure; the server's `validateAutomationDefinition` re-checks on persist (it also catches ORPHANs the lightweight local guards intentionally don't). **UI — placement mode:** `AutomationBuilder` holds `placement: {op:'move'|'duplicate', rootId} | null`; choosing Move/Duplicate enters it. `AutomationCanvas` then (a) shows a sticky `placement-banner` ("Select where to move the branch — pick a ＋" / "…place the copy…") with a `placement-cancel` button, (b) re-labels every `+` to `placement-target` routing to `onPickTarget(edge)` instead of `onInsert`, (c) for MOVE, HIDES the `+`s inside the moving subtree (invalid destinations) via `subtreeNodeIds(model, rootId)`, (d) cancels on **Escape** (a document capture-phase listener). **RICH WAIT-UNTIL — combinable time gate + condition gate + max-wait cap (v0.76.0, extends §9B; NO migration — `definition`/`automation_enrollments.state` are jsonb):** the WAIT-UNTIL node (the `wait_until` palette type — a `type:'wait'` DSL node distinguished from the SIMPLE relative `delay` wait by carrying ANY of the gate fields below) became a flexible "hold until conditions are met" node. The SIMPLE `wait` (`delay` only) is UNCHANGED.
- **DSL** (`services/automation-runner/src/dsl.ts`): `WaitNode` gained optional `untilOffset {amount, unit('minutes'|'hours'|'days'), anchor, direction?}` (relative TIME gate — `anchor` is `'now'` OR a `{{...}}` token EXPRESSION resolving to a timestamp; `direction` is `'after'` (default — ADDS the offset, e.g. `1 day from {{event.appointment_at}}`) or `'before'` (SUBTRACTS — e.g. `4 days BEFORE {{event.appointment_at}}`, the reminder pattern)), `waitCondition` (a §8 `AstNode` CONDITION gate, the SAME segment compiler/RuleBuilder), `combine: 'and'|'or'` (how the time + condition gates combine, default `'and'`; only meaningful when BOTH present), and `maxWait {amount, unit}` (a CAP). `validateAutomationDefinition` requires ≥1 of `delay|until|untilOffset|waitCondition|maxWait`, validates the duration units/amounts (`validateWaitDuration`, units whitelisted; an expression anchor must be a non-empty string), rejects a `combine` other than `'and'`/`'or'`, and rejects an `untilOffset.direction` other than `'before'`/`'after'`. The existing absolute `until` (tz-aware) stays. `isWaitUntil(node)` = has any of until/untilOffset/waitCondition/maxWait.
- **Semantics (the user's design — "both combinable", "proceed on timeout", "re-check every sweep"):** PROCEED to `next` when the TIME and CONDITION gates are satisfied per **`combine`** (`'and'` — both required; `'or'` — either), among ENABLED+resolvable gates only — OR when `maxWait` elapses (the cap is ALWAYS an OR / proceed-on-timeout, **single output edge**, no timeout branch). `combine` matters only when BOTH gates are present (an unresolvable `{{anchor}}` drops the time gate → governed by the condition; a no-gate-no-deadline node proceeds rather than stranding). A pending condition is **re-checked every sweep** (parked at `now`; for OR the time target is ALSO a wake candidate so either gate can fire), bounded by the cap.
- **Runner** (`core.ts` `decideRichWait` — PURE; `run.ts` resolves the inputs): on FIRST arrival the absolute time TARGET (resolved `until`/`untilOffset`; an expression anchor rendered via `renderExpression` over the customer/event/journey merge → a `Date`; an UNRESOLVABLE anchor DROPS the time gate) + the max-wait DEADLINE (`now + maxWait`) are computed and **PINNED on `state.wait.<nodeId> = {target, deadline}`** so a later sweep doesn't drift against a moving `now`/re-resolved anchor. The condition is evaluated like a `condition` node (`buildBranchMatchQuery`, trigger-event leaves folded first). The pin is persisted ATOMICALLY in the guarded advance UPDATE (`buildAdvanceEnrollment` gained an optional `waitPin` statePatch — a SEEDED nested `jsonb_set` on `state.wait`, the node id a BOUND jsonb-path param; folded into the SAME update so it doesn't bump `updated_at` separately and break the claim guard). Idempotent + single-winner FOR UPDATE unchanged.
