## What is being built

A **serverless, multi-tenant marketing CDP** on AWS: companies get isolated workspaces, ingest behavioral events over HTTP, maintain unified customer profiles with rolling aggregates, define segments, and reach people via broadcasts (one-off blasts) and campaigns (multi-step journeys) through Amazon SES. An admin SPA provides segment/campaign builders, a WYSIWYG email editor, dashboards, and per-workspace cost views.

## Tech stack (all decisions are locked — see §0)

- **Backend:** TypeScript on Node.js 20+, AWS Lambda (real Node runtime, full npm incl. `sharp`).
- **HTTP entry:** API Gateway **REST API** (not HTTP API — chosen for request validation, per-workspace API keys + usage plans, native WAF).
- **Database:** Supabase (managed PostgreSQL), pooled connection from Lambda. System of record.
- **Queue:** SQS **FIFO** (+ DLQ), `MessageGroupId = profile_id`.
- **Email:** Amazon SES; each workspace sends from its own verified domain.
- **Auth:** Supabase Auth; validated at the gateway by a **Lambda authorizer** (REST API has no native JWT authorizer).
- **Email editor:** a CUSTOM in-house designer (`web/src/email-designer/`, ported from the owner's nomentor builder; Preact + @preact/signals) — this deliberately OVERRIDES the spec's GrapesJS choice. The surviving invariant is unchanged: the editor **emits MJML, never hand-rolled HTML** (design JSON → `mjml-serializer.ts` → server `compileMjml`, strict). Templates are a LIBRARY; attaching one to a broadcast/campaign CLONES it (`kind='copy'`, `source_template_id`) into an independently editable working copy.
- **IaC:** AWS CDK (TypeScript).
- **Frontend:** Preact/React SPA, Vite, static on S3 + CloudFront. **Server-calling buttons auto-lock:** the shared kit `Button` (`web/src/ui/kit.tsx`) shows a spinner + disables itself whenever its `onClick` returns a Promise — so wire every server-calling button so its handler RETURNS the promise (`onClick={save}` / `onClick={() => save(id)}`, never `() => void save()`), and pass `loading` explicitly for `type="submit"` buttons. This is a standing rule (no double-submits, clear in-flight state).
- **Testing:** Vitest or Jest; Supabase CLI / Testcontainers for ephemeral Postgres; `aws-sdk-client-mock` for AWS SDK; LocalStack for the thin E2E tier.

## Non-negotiable invariants

These are the properties the whole system exists to guarantee. Every change must preserve them; they are pass/fail acceptance gates (§18), not aspirations.

1. **Tenant isolation.** Every tenant-scoped row carries `workspace_id NOT NULL`. Isolation is defense-in-depth: (a) app code filters by `workspace_id` in *every* query, (b) Postgres RLS for user-context (admin app) connections. **Critical caveat (§3):** backend processing Lambdas connect with the Supabase **service role, which bypasses RLS** — they MUST scope by `workspace_id` in code; RLS is not their safety net.

2. **Workspace is never client-supplied.** Derive it from the API key at ingest (§7) and from the authorizer-injected claim on the admin API (§12). Never trust a `workspace_id` in a request payload.

3. **Per-profile event ordering.** `MessageGroupId = profile_id` gives per-profile FIFO. Code must be **idempotent and order-convergent** so a `progress` arriving before `profile_created` still resolves correctly (upsert a stub). Don't unit-test "the queue delivered in order" — that's SQS's job; test your convergence.

4. **No lost events.** `200` is returned only after SQS accepts (durable boundary). Forced failures retry, then go to DLQ — never vanish.

5. **Idempotency.** `events.event_id` is producer-supplied and the dedupe key (`INSERT ... ON CONFLICT DO NOTHING`); `outbox.dedupe_key` and broadcast `(broadcast_id, profile_id)` prevent double-sends.

6. **Segment SQL is compiled, never interpolated.** The rule-AST → SQL compiler (§8) whitelists fields/operators, emits parameterized SQL only, and **always** prepends `workspace_id = $ws`. This is security-critical and the single highest-value unit-test target.

7. **Sending is gated on verification.** The Dispatcher refuses to send for any workspace not `active`/verified (§10). Every send passes suppression → frequency-cap → quiet-hours checks. **Verification source (overrides §10A):** the gate (`canSend`) is fed from a VERIFIED row in `sending_domains` — the per-domain model's source of truth — derived in `services/dispatcher/src/dispatch.ts` (a verified domain also supplies the `no-reply@<domain>` From fallback). The legacy `workspaces.sending_identity` jsonb is only a fallback for old data; the per-domain onboarding flow never populates it, so gating on it alone silently refused fully-verified workspaces (regression test: `services/dispatcher/test/verified-domain-gate.integration.test.ts`).

## Roles (§3A)

Four roles. `system-admin` (in `platform_admins`) is the **only cross-tenant role** — the authorizer injects an `is_platform_admin` claim, RLS has a narrow exception for it, and every cross-tenant access is written to `admin_audit_log`. The other three (`owner` / `marketer` / `accounting`) are workspace-scoped via `workspace_users.role`; a user may hold different roles in different workspaces.

## Architecture flow

```
Producer --API key--> API GW REST (validate, usage plan, WAF)
  --> Ingest Lambda (resolve workspace from key, upsert profile) --> SQS FIFO (group=profile_id)
  --> Processor (idempotent, workspace-scoped): events -> profile_features -> re-eval segments -> membership diff + change_log
        -> segment entry drives campaign enrollment (§9B) and is available to broadcasts (§9A)
  --> outbox row -> 2nd SQS queue --> Dispatcher: suppression -> freq-cap -> quiet-hours -> SES (workspace config set) -> messages_log
SES events -> SNS -> Feedback Lambda -> per-workspace suppression + reputation policing (auto-suspend offender)
```

Broadcasts (§9A) and campaigns (§9B) both emit sends through the **same** outbox → Dispatcher → SES pipeline. Campaigns are a **table-driven state machine** over `campaign_enrollments`, swept by a scheduled Campaign-runner Lambda (chosen over Step Functions: cheaper for long multi-day waits at this scale).

**Lambda handler pattern (§21):** keep each handler thin — it wires up a **pure, injected function** that holds the logic, so logic is unit-testable without the AWS/Lambda wrapper. Every handler is idempotent, stateless, and workspace-scoped.

## How to build (until tooling exists, this is the contract)

- **Build tenancy first** (workspaces, `workspace_id` everywhere, RLS, authorizer) before any feature, so nothing is retrofitted.
- **Work phase by phase (§17)**; a phase is done only when its §18 acceptance criteria pass as tests in CI.
- **Test-first (§16A):** write the phase's §18 criteria as failing tests, then implement to green.
  - **Don't mock Postgres** in the integration tier — ordering, idempotency, and isolation bugs live in the SQL and RLS. Use local Supabase (Supabase CLI) or Testcontainers.
  - **Do mock SES** — assert `SendEmail` is called with the right Configuration Set, only after suppression/cap checks; never send real mail.
- Local dev: SAM local / serverless-offline + LocalStack (SQS/S3/SNS/API GW) + Supabase CLI. Seed multiple workspaces with overlapping `external_id`s/emails to test isolation.

### Status of scaffolding

**Company → Workspace hierarchy (extends spec §6):** a `companies` table groups workspaces — every `workspace` has `company_id NOT NULL` (migration `0012`). Tenant **isolation is unchanged**: it stays at the workspace level (every tenant row carries `workspace_id`; RLS still keys on `workspace_id`). A company is purely the organizational parent so a platform admin can pick a company → a workspace within it (sidebar `CompanyWorkspacePicker`, fed by `GET /admin/companies`). Workspaces created without a `company_id` (chiefly in integration tests) are auto-assigned to a shared `Unassigned` company by a BEFORE-INSERT trigger (migration `0013`); the product's own creation paths (registration, `POST /workspaces`) always supply a real `company_id`, so `Unassigned` never appears from normal use. **A user belongs to ONE company:** access is per-workspace (`workspace_users`), but `addMember` rejects adding a user to a workspace whose company differs from one they're already in (409) — so a user's memberships always stay within a single company (which may own several workspaces).

**Registration creates a COMPANY only — never a workspace (the owner creates the first one manually).** `POST /auth/register` (`registerOwner`, local-api `session.ts`) inserts the owner `users` row + a `companies` row tagged with `owner_user_id` (migration `0032`) — NO workspace, NO `workspace_users`. The minted token is logged-in but workspace-less, so the response carries `needs_workspace: true` + the `company`. Because the production `authorize()` core hard-denies a non-platform-admin token with no active workspace, the workspace-less owner can't reach any capability-gated route (not even `/me`); so creating the first workspace is a **session route** `POST /workspace/bootstrap` (`createFirstWorkspace`) — like `/workspace/switch`, it authenticates the token directly, resolves the company server-side from `owner_user_id = <token sub>` (never client-supplied, inv.2), inserts the workspace + owner `workspace_users`, and **re-mints the token** with the new active workspace. `devLogin` mirrors this: a workspace-less owner who owns a company logs in 200 with `needs_workspace` (instead of the old 403); anyone else with no membership is still 403. SPA: `App.tsx` routes a logged-in non-admin with `!workspaceId` to the `CreateFirstWorkspace` screen (session `needsWorkspace`); `restoreSession` skips `/me` in that state; `bootstrapWorkspace()` enters the shell. Additional workspaces (for an owner already in one) are still created via the capability-gated `POST /workspaces` in Company settings.

**Identity key (overrides spec §6/§7):** a profile's per-workspace identity/merge key is **`email`**, NOT `external_id`. Events arrive from many source systems; email is the only identifier that stitches a person's events together, so ingestion requires `email`, profiles are `UNIQUE(workspace_id, email)` (migration `0010`), and the ingest/processor upserts + the manual `POST /profiles` all merge on `(workspace_id, email)`. `external_id` is optional metadata (the spec's original dedupe-by-external_id design is superseded).

**Sending domains are a LIST with a per-domain setup screen (extends §10):** a workspace can have several sending domains — `sending_domains` table (migration `0021`, workspace-scoped + RLS), each with a `verified` flag. UI is `/onboarding` (the **list only** — `SendingDomainsList`, nav label "Sending domains") → click a row or "Add domain" opens the **per-domain setup screen** `/onboarding/:id` (or `/onboarding/new`) — `SendingDomainDetail`. There you save the domain (pending), see its **Amazon SES Easy-DKIM CNAME records** to publish, and **verify via SES** (`POST /sending-domains/:id/check`). Verification uses **real SES**: on first open each domain is provisioned as an SES email identity (`SesEmailClient.createDomainIdentity` → 3 DKIM tokens, stored in `sending_domains.ses_identity` + `dkim_tokens`, migration `0023`); the check calls `getIdentityVerificationAttributes` and verifies ONLY when `DkimStatus === 'SUCCESS'` — no manual flip. **The DKIM CNAME target host is read FROM SES, never constructed** (multi-region: each company picks its own SES region): SES returns `DkimAttributes.SigningHostedZone` (region-specific, e.g. `dkim.il-central-1.amazonses.com`), persisted in `sending_domains.signing_hosted_zone` (migration `0029`) and used verbatim as `<token>.<signing_hosted_zone>` (`dkimCnameHost` in `handlers.ts`; pre-`0029` rows backfill on next load). Fallbacks only if SES omits it / local mock: `dkim.<region>.amazonses.com`, then `dkim.amazonses.com`. **SES is PER-COMPANY:** each company stores its own AWS SES credentials (`company_ses_config` table, migration `0024`: region + access_key_id + secret_access_key; secret is write-only over the API, never returned; company-scoped RLS like `companies`). Managed in **Company settings** (`CompanySesConfig`, API `GET/PUT/DELETE /company/ses-config`, capability `manage_sending_domain`). The sending-domain handlers build the SES client from the active workspace's company config (`sesForWorkspace` → `createSesClient` from `@cdp/email`); **with no company config they fall back to the local MOCK** SES, so dev/tests verify deterministically and the domain setup screen shows a "verification is simulated" note (`sesConfigured:false`). (The earlier `USE_REAL_SES` env switch and the `0022` TXT-token approach were superseded; the `verify_token` column was dropped in migration `0030`.) A `domain_senders` row (named "From" identity, migration `0020`) may only be created for a **verified** domain (enforced in `createDomainSender`) and is managed **inside that domain's setup screen** (local-part input fixed to `@<that-domain>`), never in a global list. API (all `manage_sending_domain`): `GET/POST /sending-domains`, `GET /sending-domains/:id` (domain + DNS records), `POST /sending-domains/:id/check`, `DELETE /sending-domains/:id` (blocked while it has senders); `GET/POST /domain-senders`, `DELETE /domain-senders/:id`. (The legacy workspace-level wizard routes `/sending-domain/{start,check,activate}` were removed — the per-domain screen above replaces them. The onboarding cores `startDomain`/`checkDomain`/`activate` in `@cdp/service-onboarding` remain, exercised by the acceptance/integration tiers.) **`GET /domain-senders` is `manage_content`** (read needed by the broadcast/campaign From dropdown); creating/deleting a sender stays `manage_sending_domain`.

**From / To / Subject live on the EMAIL INSTANCE (a broadcast/campaign's working copy), NOT on a library template (migration `0028`):** `email_templates` gained `subject text`, `sender_id uuid → domain_senders(id) ON DELETE SET NULL`, and `to_address text NOT NULL DEFAULT '{{customer.email}}'`. A **library template is just a reusable DESIGN and shows NO envelope** in the editor — the From/To/Subject Card (`email-sender` / `email-to` / `email-subject`) renders ONLY when the editor is editing an email *instance* (`TemplateEditor`'s `instance` = opened via a broadcast/campaign return flow OR `kind='copy'`). Attaching a template to a broadcast/campaign **clones** it into a working copy (`kind='copy'`), and that copy's envelope is edited in the editor (autosaved with the design); `cloneTemplate` copies these columns. **Editor return context** (`store/editorReturn.ts`) is persisted in `sessionStorage` so a page refresh inside the editor keeps the contextual "← Back to broadcast/campaign" target (it used to fall back to the template library); standalone editor opens (TemplatesList "New template"/Edit, the "Design email" header shortcuts) call `clearEditorReturn()` so a stale return can't mislabel Back. The **To** is a recipient token (default `{{customer.email}}`) rendered per recipient at send (falls back to `profile.email`; suppression/unsubscribe still key on `profile.email`). **From / To / Subject are ALL MANDATORY to send a broadcast — there is NO no-reply fallback (overrides the earlier `from_selected` design).** The **From** must be a real named sender (a verified-domain `domain_senders` row, `email_templates.sender_id`); the editor's From dropdown has only a disabled "Choose a sender…" placeholder + the named senders (NO "No-reply" option), and shows a hint to add one under Sending domains when the workspace has none. `sendBroadcast` joins the template and 409s in order: "Choose who the email is from…" when `sender_id` is null, "Set the To field…" when `to_address` is blank, "Add a subject line…" when `subject` is blank — *then* the verified-domain gate. The **Dispatcher** reads `compiled_html, subject, sender_id, to_address` from the template, resolves `sender_id` → `"Name" <email>` (`fromAddress` in `core.ts`, which still has a `no-reply@<domain>` last-resort for a null `sender_id` — broadcasts can't reach it), and renders `to_address`. The broadcast/campaign **outbox payload no longer carries** subject/sender — only `broadcast_id`. `cloneTemplate` copies `subject`/`sender_id`/`to_address`/`from_selected` onto the working copy (so attaching a configured library template yields a sendable copy); the e2e seed creates a `domain_senders` row and points the seeded template's `sender_id` at it so seeded-template clones stay sendable. `validateSenderId` (local-api, reused by `createTemplate`/`updateTemplate`) rejects a cross-workspace `sender_id` (inv.2); `GET /domain-senders` is `manage_content` (the editor's From dropdown). The broadcast wizard / campaign builder no longer have envelope fields — they point you to "Design email". **The wizard also gates its Content step on the envelope:** it fetches the attached copy's `{subject, sender_id, to_address}`, computes `missingEnvelope` (From/To/Subject), and `step1Valid` stays false until all three are set — so `wizard-next` and the later breadcrumbs (`canReach`) are disabled and an `email-incomplete` hint lists what's missing (`email-complete` once done). The user can't reach Schedule/Send without a deliberately-chosen From. (`from_selected` migration `0034` remains in the schema but is no longer the gate — `sender_id` presence is.) **Broadcast Content step (instance model):** until an email exists the step offers "Start from a template" (a picker that CLONES the chosen library template into the broadcast's own copy on select) OR "Start from a blank design" (`design-email` → editor, `createAs:'copy'`). Once a copy is attached the picker is GONE — the step shows the instance (`email-instance`) with **Edit email** (`design-email`) and **Start over** (`email-replace`, which just unreferences the copy). The template is only a starting point; you cannot swap the underlying template inline. Returning from the editor lands on the **Content** step (the editor records `markReturnedTo`; the wizard restores the step even when no copy was saved). The **Schedule** step offers **Send now** (the default — the finish button SENDS immediately via `POST /broadcasts/:id/send`, surfacing a send failure rather than leaving a silent draft) or **Schedule for a date & time** (saves a `scheduled` broadcast), plus a separate **Save as draft** action (`wizard-save-draft` → persists with no send time, status `draft` — "created but not sent or scheduled"). Scheduling also offers a **timezone** (`schedule-tz`, default the browser zone, full IANA list via `Intl.supportedValuesOf`): the wall-clock time is interpreted in that zone → UTC (`zonedInputToUtcIso`, DST-correct) and the chosen zone is persisted in `broadcasts.scheduled_tz` (migration `0033`) so edit round-trips it and the list shows the send time in that zone (`whenLabel(ts, tz)`). **A scheduled send must be ≥ 5 minutes from now** (`MIN_SCHEDULE_LEAD_MS`, enforced in BOTH places): the wizard sets the picker's `min`, disables `wizard-save`, and shows a `schedule-too-early` hint when the chosen instant is sooner; the server's `createBroadcast`/`updateBroadcast` re-check via `scheduleLeadError` and 400 ("A broadcast must be scheduled at least 5 minutes from now."). The step **breadcrumbs are clickable** — `canReach(target)` lets you jump to any step whose prerequisites are met instead of clicking Next. **Row actions on the broadcasts list are consolidated into one kebab (⋮) `ActionMenu`** (`broadcast-actions` trigger; reusable component in `web/src/ui/kit.tsx`): Edit/Send/Delete only for unsent (draft/scheduled), Duplicate always; an async item keeps the menu open + spins itself + locks the others until it settles (the standing server-button rule). Each item keeps its original `data-testid` (`broadcast-edit`/`broadcast-duplicate`/`send-broadcast`/`broadcast-delete`) — e2e tests just open the menu first. (The **campaign builder is still a placeholder** that auto-attaches `templates[0]` with no email-choice UI — the same instance flow will apply once its send-node config UI exists.)

**`{{unsubscribe}}` body token + two-step opt-out (§10):** the dispatcher resolves two merge tokens per recipient — `{{unsubscribe}}` (a ready `<a>…</a>`) and `{{unsubscribe_url}}` (the raw URL, for a custom-text link) — to the workspace-scoped link `…/unsubscribe?workspace_id=&email=` (`@cdp/email buildUnsubscribeUrl`, added to the merge map in `services/dispatcher/src/dispatch.ts`; `renderTemplateBody` does not escape merge values, so the anchor renders). The link is **deliberately two-step**: a **GET** (the click from the email) returns a re-affirm CONFIRMATION page and changes NOTHING — mail clients/proxies prefetch GET links, so a GET must never opt anyone out; only a **POST** (the page's "Yes, unsubscribe" button, or an RFC 8058 one-click) writes the per-workspace `suppressions` row AND sets `profiles.attributes.unsubscribed = true` (`buildUnsubscribeSuppression` + `buildUnsubscribedAttribute`, ONE workspace-scoped tx). The SAME `@cdp/service-unsubscribe` handler (`makeUnsubscribeHandler`, now serving HTML) backs both the production Lambda AND the local-api public `GET/POST /unsubscribe` routes (`app.ts`). (Local caveat, like images: the link points at `unsubscribeBaseUrl` = `<app base>/unsubscribe`, which is localhost in dev — reachable from your browser, not a remote recipient.)

**Activity log = events + email feedback + sends + `activity_log` (migration `0035`):** `GET /activity` (`listActivity`) UNIONs four sources. The first three are `events` (behavioral), `email_events` (SES feedback), `messages_log` (sends). The fourth, `activity_log`, is an append-only feed for SYSTEM/admin actions that aren't behavioral events and must NOT pollute `events` (which feeds segment rules + the profile's behavioral timeline): a recipient **unsubscribing** (`source='unsubscribe'`, written in the unsubscribe tx via `buildUnsubscribeActivity`), a **profile created/updated** (`source='profile'`, type `profile_created`/`profile_updated`, written by `createProfile`/`updateProfile`), and a **CSV import** (one summary `profiles_imported` row, not per-profile). `activity_log.workspace_id` is `ON DELETE CASCADE` (a workspace's activity dies with it, so test teardowns that just delete the workspace need no extra step) and `profile_id` is `ON DELETE SET NULL` (+ reassigned to the survivor on profile merge). **Scheduled broadcasts show a live countdown** on the list ("Scheduled to send … · in 2 days, 5 hours" — `untilLabel`, re-rendered each minute). **They actually fire via a sweep:** production runs `@cdp/service-broadcast` `scheduledSweepHandler` on an EventBridge cron; the **dev server has no scheduler**, so `services/local-api/src/server.ts` runs an in-process sweep every `LOCAL_SCHEDULE_SWEEP_MS` (default 30s) calling `sweepDueScheduledBroadcasts(pool, deps)` — for each broadcast whose `scheduled_at` has passed it runs the broadcast + `dispatchBroadcastNow` (so it sends for real when the company has SES creds), mirroring the manual Send. The sweep runs ONLY in the long-lived dev server, never in `createApp` (so tests don't get a background timer); the `0c0d…`-scoped `scheduled-sweep.integration.test.ts` covers the core.

**Profile actions menu + manual "Send event" (extends §7/§12):** the Profile detail header consolidates its actions into one kebab (⋮) `ActionMenu` (`profile-actions`, the same reusable `ui/kit.tsx` component as the broadcasts list) — **Send event** (`send-event-action`) and **Merge…** (`merge-button`, moved in from a standalone button). **Send event** (`SendEventDrawer`) records a SINGLE behavioral event on the profile's behalf: a `type` (autocompletes from `/events/types`) + an optional JSON `payload` (validated client-side as a JSON object). `POST /profiles/:id/events` (`sendProfileEvent`, `manage_content`) inserts an `events` row (`gen_random_uuid()`, `occurred_at = now()`) and, in ONE workspace-scoped tx, recomputes the profile's rolling features + re-evaluates its dynamic_realtime segments via the shared `recomputeFeaturesAndSegments` helper (extracted from `mergeProfiles`, which now also calls it). So a manual event behaves exactly like an ingested one — it feeds segment rules, the timeline (the Events tab remounts to show it), and the features (header stats refresh). Rejects a missing `type` or a non-object `payload` (400); a cross-workspace profile id is 404 (inv.1/2).

**Broadcast metrics + click tracking (extends §9A/§10):** the broadcasts list shows per-broadcast **Failed / Delivered / Clicked** (Opened & Converted deferred). Attribution: `messages_log.broadcast_id` (migration `0026`, mirrors `campaign_id`; the dispatcher writes it from `outbox.payload.broadcast_id`). Delivered/Failed are aggregated from `email_events` (delivery/bounce/complaint) joined to `messages_log` by `ses_message_id`; Clicked sums `tracked_links.clicks`. **Click tracking is opt-in per workspace** (`workspaces.settings.link_tracking`, toggled in Workspace settings). When on, the **dispatcher** (the single point ALL sends pass through — broadcasts AND campaigns) rewrites every `http(s)` link in the email to `<linkTrackingBaseUrl>/t/<token>` (`rewriteTrackingLinks`, deterministic sha256 token per `(workspace, broadcast|campaign, url)` → idempotent + shared across recipients) and upserts `tracked_links` rows (migrations `0026`/`0027`). The public `GET /t/:token` (in `app.ts`, like `/assets/:id`) 302-redirects to the original URL and increments `clicks`. Metrics are 0 in local dev (the dispatcher/feedback pipeline doesn't run); they populate in a deployed pipeline. A `broadcast.updated_at` drives the "Edited X ago" line.

**Workspace delivery-health on the dashboard (extends §10):** the Dashboards screen (`SimpleScreens.tsx`) has a **Delivery health** section fed by `GET /dashboards/delivery-health?days=N` (`manage_content`, default 30d, clamped 1..365). It returns, all workspace-scoped: `outcomes` (sent from `messages_log`; delivered/bounced/complained from `email_events`, windowed by `occurred_at`), `rates` (bounce = bounced/(delivered+bounced); complaint = complained/delivered) colored against SES thresholds (bounce >5% warn / >10% danger; complaint >0.1% warn / >0.5% danger, shown via `rateTone`), `suppression` (current list size by reason — NOT windowed), and a gap-filled per-day `trend` (`generate_series`) rendered by a dependency-free `Sparkline` (`ui/kit.tsx`). Same caveat: outcomes/rates are 0 in local dev (no feedback pipeline); the suppression count is live. testids: `delivery-health`, `dh-sent/delivered/bounced/complained`, `dh-bounce-rate`, `dh-complaint-rate`, `dh-reputation-warning`, `dh-suppressed-total`, `dh-supp-<reason>`, `dh-trend`.

**`customer.*` personalization namespace (extends §8/§11):** one systemwide token scheme for referring to a profile in segment rules AND email merge tags. `customer.email` / `customer.external_id` / `customer.email_status` / `customer.created_at` map to the matching `profiles` column; `customer.attributes.<key>` is a custom attribute; and **`customer.<key>` is shorthand for `customer.attributes.<key>`** (so `{{customer.tier}}` ≡ `{{customer.attributes.tier}}`, and in a segment rule `customer.tier` ≡ `attributes.tier`). The single source of truth is `packages/shared/src/customer.ts` (`expandCustomerPath` / `expandCustomerToken` / `resolveCustomerField` / `customerMerge`). The segment compiler (`packages/segments`) normalizes the field BEFORE the whitelist (invariant 6 unchanged); the dispatcher builds the `customer.*` merge from the recipient profile and `renderTemplateBody` expands the shorthand at send time. **Merge tags are rendered in the SUBJECT, the To, AND the HTML body** — all three pass through `renderTemplateBody` in `buildSendEmailInput` (`services/dispatcher/src/core.ts`); the subject used to be sent raw (bug fixed v0.27.2 — `{{customer.attributes.first_name}}` in a subject now resolves per recipient). Legacy bare `attributes.*` / scalar field names still work.

**Workspace timezone + campaign DSL extensions (extends §9B; campaign-builder phase 1):**

- **Workspace timezone** is persisted on `workspaces.settings.timezone` (jsonb — NO migration) and read/written through the existing `GET/PUT /workspace/settings` (owner-gated, capability `manage_workspace_users`). It **defaults to `UTC`** (`getWorkspaceSettings`), is **validated as a real IANA zone** server-side before write (`isValidTimeZone`, 400 on an invalid zone — nothing written), and `workspace_id` is taken from ctx only (a stray body `workspace_id` is ignored, inv.2). The jsonb merge preserves siblings (`lowercase_emails`/`link_tracking`). UI: a small IANA `Select` picker in `web/src/screens/WorkspaceSettings.tsx` (testids `workspace-timezone-select` / `workspace-timezone-save`), fed by `timeZoneList()`, optimistic save with rollback + `showToast` on failure (the save handler RETURNS the promise so the kit Button auto-locks; pure logic in `web/src/screens/workspaceSettingsLogic.ts`). **This is the single clock for ALL campaign time math** (waits, wait-until, hour-of-day windows) — DST-correct, never per-broadcast guesswork.
- **Shared timezone helpers** — `tzOffsetMs` / `zonedInputToUtcIso` / `utcIsoToZonedInput` / `timeZoneList` + a new `isValidTimeZone` now live in **`packages/shared/src/timezone.ts`** (exported from the `@cdp/shared` barrel), EXTRACTED from `BroadcastComposer.tsx` (which re-imports them) so the broadcast scheduler and campaign time math share ONE DST-correct implementation. `timeZoneList()` guarantees `'UTC'` is present (some engines list only `Etc/UTC` in `Intl.supportedValuesOf`).
- **Two new campaign DSL node kinds** (`services/campaign-runner/src/dsl.ts`, re-exported via `index.ts`) — **TYPES + STRUCTURAL validation only this phase; NO runner execution (phase 2)**: (a) an **`hour_of_day_window`** node (`startHour`/`endHour` integers 0–23, optional `daysOfWeek` 0–6 unique non-empty; `startHour > endHour` is a VALID overnight wrap-around; `next` edge) — the runner will later PARK an enrollment until the next window opening using the workspace tz; for now `processNode` advances it as a no-op. (b) a **`webhook`** action kind on the action node (`url` http(s) ONLY — a model-layer SSRF pre-check; `method` ∈ GET/POST/PUT/PATCH/DELETE; optional `headers` object, `bodyTemplate` string, positive `timeoutMs`, non-negative integer `maxRetries`; `next` edge) — phase 2 adds the injected HTTP client, a per-workspace host allowlist + SSRF runtime guards, bounded retries, and envelope-encrypted (`@cdp/db` secret-crypto) auth-header secrets; for now `actionSideEffect` returns null for it.
- **`validateCampaignDefinition` is tightened to the builder invariants** (the DSL stays the single graph model): in addition to the existing exactly-ONE-trigger / resolvable-edges / reachable-exit / per-type-field checks, it now **rejects CYCLES** (`detectCycle` — a DFS grey-stack; a self-loop or a back-edge to a node on the current recursion stack throws, but a diamond / re-convergence is ALLOWED) and **rejects ORPHANS** (`findOrphans` — every defined node must be reachable from `startNode`). Cycle detection runs BEFORE the reachable-exit check so a looping graph is diagnosed as a cycle rather than a generic "no exit".

Phase-1 scaffolding only — no business/feature logic yet.
